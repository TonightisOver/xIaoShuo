# 项目文件归档与工程结构优化设计

> 在不改变业务逻辑、接口行为、数据模型和页面功能的前提下，先把历史工程资料集中归档，再分阶段整理后端、前端和测试目录。

## 背景

当前仓库共有约 827 个 Git 跟踪文件，其中 `.harness` 历史流程资料约 397 个，已经超过业务源码文件数。除此之外，本地工作区还包含前端依赖、构建产物、Python 缓存、测试缓存、日志和 Claude 工作树。业务源码本身已经经过多轮拆分，现有 `API → Service → Core` 依赖方向和领域服务边界不能在本次整理中被破坏。

本次优化解决的是文件可发现性、活动目录噪声、历史资料检索和后续分包秩序，不解决业务逻辑、性能、UI、数据库结构或功能缺陷。

## 目标

1. 在项目根目录建立统一的 `archive/` 归档区。
2. 从历史资料中提取目标、设计、实现、验证和结论，形成可搜索的归档记录。
3. 将已完成的历史资料移出活动工程目录，同时完整保留原文和 Git 历史。
4. 清理可再生成的缓存与构建产物，不删除依赖、日志、数据或环境配置。
5. 建立移动前后路径清单，确保整理可审计、可验证、可回退。
6. 第一阶段稳定后，再分别规划后端、前端和测试目录分包。

## 非目标

- 不修改 Python、Vue 或 JavaScript 业务逻辑。
- 不改变 API 地址、请求响应、数据库模型或迁移历史。
- 不拆分大型源码文件，不顺带修复代码质量问题。
- 不删除 `frontend/node_modules/`、`logs/`、`data/`、`.env` 或 `poetry.lock`。
- 不移动 `.claude/worktrees/`，也不处理其中已有修改。
- 不在同一批次内同时执行历史归档和源码分包。

## 方案选择

### 方案一：只移动历史资料

风险最低，但只能减少活动目录噪声，不能为后续源码增长建立明确秩序。

### 方案二：一次性重排整个仓库

目录变化最明显，但会同时产生大量文档链接、Python import、Vue import、测试路径和构建配置变更，回归定位困难。

### 方案三：分阶段系统整理（采用）

先完成历史资料归档和工作区清理；验证通过后，把后端、前端和测试分成独立子工程逐项设计、实施和验证。每个阶段都有独立提交和回退边界。

## 总体阶段

### 第一阶段：历史资料归档与工作区清理

本设计文档和下一步实施计划只覆盖第一阶段。

### 第二阶段：后端源码分包

保持 `API → Service → Core` 依赖方向，只在现有层内按领域建立子包。该阶段需要单独设计路径映射和兼容策略。

### 第三阶段：前端与测试分包

前端按功能领域归类页面、组件和 composable；测试按 `unit / integration / regression` 与业务领域归类。该阶段单独设计，不与后端移动混合。

## 第一阶段目录设计

```text
archive/
├── README.md                 # 归档用途、规则和查阅方式
├── baseline.json             # 整理前活动文件统计
├── index.md                  # 面向人的归档总索引
├── manifest.csv              # 旧路径、新路径、类别和状态映射
├── organization-report.md    # 整理前后结果与保留项
├── changes/                  # 已完成的 Harness 变更资料
├── plans/                    # 已完成的实施计划
├── specs/                    # 已完成的设计规格
├── reports/                  # 零散交付、验证和变更说明
└── scratch/                  # 已完成的临时 issue/map 资料
```

`archive/` 纳入 Git 管理。归档不是备份压缩包，而是仓库内的只读历史资料区。

## 归档范围

### 纳入归档

- `.harness/changes/CHANGE-*` 中已经完成的变更目录。
- `docs/superpowers/plans/` 中已经完成或已被提交实现的旧计划。
- `docs/superpowers/specs/` 中已经完成或已被后续计划承接的旧规格。
- `.scratch/` 中状态为完成、解决或仅供历史追踪的资料。
- `docs/` 中明确属于一次性交付、历史验证或旧变更说明的零散报告。

### 保留在活动目录

- `.harness/agents/`、`.harness/rules/`、`.harness/gates/`、`.harness/wiki/` 和模板。
- 当前归档任务的设计文档与实施计划。
- `README.md`、`CONTEXT.md`、`CLAUDE.md` 和仍在使用的指南。
- `src/`、`frontend/src/`、`tests/`、`alembic/`、`scripts/`、`loadtest/`。
- Docker、Nginx、Poetry、Vite、Alembic 和环境配置。
- 状态不明确、仍在进行或仍被当前流程直接引用的资料。

### 判定原则

若无法从文件内容、状态字段或 Git 提交确认某项资料已经完成，该项默认保留在原位置，并在归档报告中列入“待确认”，不得猜测移动。

## 归档记录格式

每个 `CHANGE-*` 目录保留全部原始文件，并新增一份 `archive-record.md`。记录采用固定字段：

```markdown
# CHANGE-NNN 归档记录

- 原名称：
- 状态：
- 时间范围：
- 原路径：
- 归档路径：
- 关联提交：

## 目标
## 主要设计决定
## 涉及模块
## 实施结果
## 测试与验证
## 遗留事项
## 原始文件清单
```

信息优先从需求分析、技术设计、编码报告、测试报告、CI 报告、最终交付文档和 `summary.md` 中提取。发生冲突时，以时间较新的最终报告和 Git 状态为准，并在“遗留事项”中记录差异，不改写原始文件。

旧计划、规格、报告和 scratch 资料不强制逐文件生成摘要；它们通过 `index.md` 和 `manifest.csv` 建立主题、状态和路径映射。

## 索引与清单

### `archive/index.md`

按类别列出归档项，至少包含编号或标题、状态、简述、归档路径和关联提交。索引只引用仓库相对路径。

### `archive/manifest.csv`

字段固定为：

```text
category,item,status,source_path,archive_path,commit,notes
```

每个被移动的文件写一行路径映射；目录级汇总只写入 `index.md`，避免 manifest 同时记录目录和文件造成重复。未移动但被检查的资料不写入 manifest，而写入实施报告的保留清单。

## 文件移动规则

- 使用文件系统移动，不复制后删除；本任务不执行 Git 暂存或提交。Git 在后续提交时可依据内容相似度识别重命名。
- 原始归档文件内容保持字节级不变；只允许新增索引和归档记录。
- 移动后允许机械更新 Markdown 链接、README 路径和流程索引。
- 不批量改写历史文档中的叙述性旧路径；只有仍作为活动入口的链接才更新。
- 不修改源码 import。源码路径变化属于后续阶段。

## 工作区清理

第一阶段允许删除以下可再生成内容：

- Python `__pycache__/` 和 `*.pyc`。
- `.pytest_cache/`、`.ruff_cache/`、`.mypy_cache/`。
- `frontend/dist/`。
- Alembic、测试和源码目录中的缓存文件。

以下内容明确保留：

- `frontend/node_modules/`。
- `logs/`。
- `data/`。
- `.env`、`.env.docker`、`.env.example`。
- `poetry.lock`。
- `.claude/worktrees/` 及其中已有修改。

清理对象必须已经被 `.gitignore` 覆盖；未被忽略的文件不得作为缓存直接删除。

## 执行流程

1. 记录 `git status`，确认用户已有修改和嵌套工作树状态。
2. 生成活动文件基线清单、文件数量和目录占用统计。
3. 分类历史资料为“归档”“保留”“待确认”。
4. 创建 `archive/` 基础目录、README、索引和 manifest。
5. 为已完成的 `CHANGE-*` 生成归档记录。
6. 使用文件系统移动分类资料，并记录完整的新旧路径映射。
7. 更新仍在使用的文档入口和相对链接。
8. 删除允许清理的缓存与构建产物。
9. 执行文档、Git、后端和前端验证。
10. 生成整理报告并提交第一阶段变更。

## 失败处理与回退

- 归档状态不明确：保留原位并记录，不阻塞其他确定项。
- 摘要信息冲突：保留原文，归档记录同时列出冲突和采用依据。
- 链接检查失败：修复机械路径；若目标文件缺失，则恢复该项移动。
- 测试或构建失败：先与整理前基线比较。若是新出现的失败，只修复路径性问题；涉及业务逻辑时停止并回退该批移动。
- 用户已有文件变化：不覆盖、不暂存、不提交不属于本任务的文件。

回退以 `manifest.csv` 的路径映射为依据。本任务不创建提交；用户后续处理 Git 时，第一阶段归档与源码分包仍应保持为不同提交。

## 验证策略

### 整理前基线

- `git status --short`
- 受管工作区文件总数与一级目录分布
- 后端测试结果
- 前端测试结果
- 前端构建结果

### 整理后验证

- `git status --short`：确认变化只涉及预期的旧路径删除、新归档文件、索引和活动文档路径更新。
- `git diff --check`：确认无格式错误。
- 检查 `manifest.csv` 的每个归档路径存在，旧路径不存在。
- 检查活动 Markdown 中不存在指向已移动旧路径的失效链接。
- `poetry run pytest tests/ -v`。
- `npm test`（在 `frontend/`）。
- `npm run build`（在 `frontend/`）。
- 验证 FastAPI 应用入口可以导入。

如果整理前基线本身有失败，整理后的通过标准是“不新增失败”，并在报告中区分存量问题和新增问题。

## Git 边界

本任务不执行 Git 暂存、提交或推送。第一阶段实施完成后，建议用户后续提交时只包含：

- `archive/` 新增内容。
- 被归档资料的移动记录。
- 必要的活动文档路径更新。
- 与清理相关且实际被 Git 跟踪的配置调整。

缓存和构建产物本身处于忽略状态，不应进入提交。

## 后续阶段约束

后端、前端和测试分包必须分别编写设计与实施计划。所有源码移动只允许机械调整 import、路由引用、测试引用和构建配置；不得夹带功能修改、抽象重构、文件拆分或命名语义变化。

每个后续阶段都必须保持外部接口、运行行为和测试语义不变，并具有独立验证和独立提交。

## 1. 项目概述

### 1.1 项目定位
AI 驱动的中文网络小说自动化创作平台，通过 DeepSeek 大模型实现从创意到成品的全流程小说生成。

### 1.2 核心价值主张
- **一键部署小说**: 从创意到完整小说的自动化生成
- **精细化控制**: 人物关系、大纲、背景、小纲、情节发展的全方位把控
- **灵活修改**: 支持对生成内容的迭代优化和人工干预

### 1.3 目标用户
- **网文作者**: 需要提高创作效率的职业/兼职作者
- **内容平台**: 需要批量生成内容的小说平台运营方
- **创意爱好者**: 有创意但缺乏写作能力的用户

## 2. 核心功能模块

### 2.1 小说创作工作流

#### 阶段 1: 创意输入
**输入内容**:
- 小说类型（玄幻、都市、科幻、历史等）
- 核心创意（一句话概述或关键词）
- 风格偏好（热血、轻松、黑暗等）
- 目标字数（短篇/中篇/长篇）

**输出**:
- 初步创意扩展方案（3-5 个方向）

#### 阶段 2: 世界观构建
**功能**:
- **背景设定**: 时代背景、世界规则、力量体系
- **地理设定**: 主要场景、地图关系
- **社会结构**: 势力分布、组织架构

**AI 能力**:
- 根据小说类型自动生成合理的世界观框架
- 检测世界观内部逻辑一致性

#### 阶段 3: 人物系统
**功能**:
- **主角设定**: 性格、背景、目标、成长路线
- **配角设定**: 关键配角的人设和作用
- **人物关系图**: 可视化的人物关系网络
  - 亲属关系
  - 师徒关系
  - 敌对关系
  - 情感关系
- **人物弧光**: 每个角色的成长轨迹

**AI 能力**:
- 根据小说类型生成典型人物模板
- 自动检测人物关系冲突
- 生成人物对话风格指南

#### 阶段 4: 大纲设计
**功能**:
- **总体大纲**: 整部小说的结构框架
  - 开端、发展、高潮、结局
  - 主线剧情
  - 支线剧情
- **章节大纲**: 每章的核心内容和目标
- **小纲（细纲）**: 每个场景的详细描述
  - 场景目标
  - 出场人物
  - 情节要点
  - 伏笔和呼应

**AI 能力**:
- 根据目标字数自动规划章节数量
- 检测剧情节奏（起承转合）
- 识别伏笔和回收点

#### 阶段 5: 内容生成
**功能**:
- **章节生成**: 基于小纲生成完整章节内容
- **场景描写**: 环境、氛围、动作描写
- **对话生成**: 符合人物性格的对话
- **情节推进**: 确保情节按大纲发展

**AI 能力**:
- 调用 DeepSeek 大模型生成高质量文本
- 保持前后文一致性
- 控制章节字数（如 3000-5000 字/章）

#### 阶段 6: 内容修改与优化
**功能**:
- **局部修改**: 修改特定段落、对话、描写
- **风格调整**: 调整语言风格、节奏
- **情节调整**: 修改剧情走向
- **人物调整**: 修改人物行为、对话
- **版本管理**: 保留修改历史，支持回滚

**AI 能力**:
- 理解修改意图，生成符合上下文的新内容
- 检测修改后的一致性问题
- 提供多个修改方案供选择

#### 阶段 7: 质量检查
**功能**:
- **逻辑检查**: 剧情逻辑、人物行为逻辑
- **一致性检查**: 人物设定、世界观设定的前后一致性
- **文本质量**: 错别字、语法、流畅度
- **节奏分析**: 剧情节奏是否合理

#### 阶段 8: 导出与发布
**功能**:
- **格式导出**: TXT、EPUB、PDF、Markdown
- **平台适配**: 适配起点、晋江等平台格式要求
- **章节管理**: 支持分章节导出

## 3. 技术架构设计

### 3.1 架构选型评估

#### 方案 A: LangGraph 架构（推荐）
**优势**:
- **状态管理**: 天然支持小说创作的多阶段状态流转
- **图结构**: 适合表达复杂的创作流程（大纲→人物→章节→修改）
- **可视化**: 流程可视化，便于调试和优化
- **检查点**: 支持中断恢复，适合长时间的小说生成任务
- **条件路由**: 根据生成质量自动决定是否需要重新生成

**适用场景**:
- 复杂的多步骤创作流程
- 需要人工干预和审核的环节
- 需要状态持久化和恢复

#### 方案 B: 传统 Pipeline 架构
**优势**:
- 实现简单，学习成本低
- 适合线性流程

**劣势**:
- 状态管理需要手动实现
- 难以处理复杂的条件分支
- 缺乏可视化和调试工具

**结论**: **推荐使用 LangGraph 架构**，理由如下：
1. 小说创作是典型的多阶段、有状态的复杂流程
2. 需要频繁的人工审核和修改（Human-in-the-Loop）
3. 需要支持中断恢复（生成长篇小说可能需要数小时）
4. LangGraph 的图结构天然适合表达"大纲→人物→章节"的依赖关系

### 3.3 LangGraph 流程设计

#### 核心状态定义
```python
class NovelState(TypedDict):
    # 项目元信息
    project_id: str
    novel_type: str  # 玄幻、都市、科幻等
    target_words: int
    
    # 创作内容
    idea: str  # 核心创意
    world_setting: dict  # 世界观设定
    characters: list[dict]  # 人物列表
    relationships: dict  # 人物关系图
    outline: dict  # 总体大纲
    chapter_outlines: list[dict]  # 章节大纲
    chapters: list[dict]  # 生成的章节内容
    
    # 流程控制
    current_stage: str
    approval_status: str  # pending/approved/rejected
    revision_requests: list[str]
    
    # 质量指标
    quality_scores: dict
```

#### 节点定义
1. **创意扩展节点** (idea_expansion)
2. **世界观构建节点** (world_building)
3. **人物设计节点** (character_design)
4. **大纲生成节点** (outline_generation)
5. **章节生成节点** (chapter_generation)
6. **质量检查节点** (quality_check)
7. **人工审核节点** (human_review)
8. **修改节点** (revision)

#### 边和条件路由
- 质量检查通过 → 下一阶段
- 质量检查不通过 → 重新生成
- 人工审核通过 → 下一阶段
- 人工审核拒绝 → 修改节点

## 4. 数据模型设计

### 4.1 核心实体

#### Project (项目)
```python
class Project:
    id: UUID
    name: str
    novel_type: str
    target_words: int
    status: str  # draft/in_progress/completed
    created_at: datetime
    updated_at: datetime
```

#### WorldSetting (世界观)
```python
class WorldSetting:
    id: UUID
    project_id: UUID
    background: str  # 时代背景
    rules: dict  # 世界规则（如修炼体系）
    geography: dict  # 地理设定
    social_structure: dict  # 社会结构
    version: int
```

#### Character (人物)
```python
class Character:
    id: UUID
    project_id: UUID
    name: str
    role: str  # protagonist/supporting/antagonist
    personality: str
    background: str
    goals: list[str]
    growth_arc: str
    dialogue_style: str
    version: int
```

#### Relationship (人物关系)
```python
class Relationship:
    id: UUID
    project_id: UUID
    character_a_id: UUID
    character_b_id: UUID
    relationship_type: str  # family/mentor/enemy/love
    description: str
```

#### Outline (大纲)
```python
class Outline:
    id: UUID
    project_id: UUID
    level: str  # main/chapter/scene
    parent_id: UUID | None
    title: str
    content: str
    order: int
    version: int
```

#### Chapter (章节)
```python
class Chapter:
    id: UUID
    project_id: UUID
    chapter_number: int
    title: str
    content: str  # 完整章节内容
    word_count: int
    status: str  # draft/generated/reviewed/published
    version: int
    created_at: datetime
```

#### Revision (修改记录)
```python
class Revision:
    id: UUID
    target_type: str  # chapter/character/outline
    target_id: UUID
    old_content: str
    new_content: str
    reason: str
    created_at: datetime
```

## 6. 非功能性需求

### 6.1 性能要求
- **章节生成速度**: 3000 字章节 < 30 秒
- **并发支持**: 支持 10+ 个项目同时生成
- **响应时间**: API 响应 < 200ms（不含 AI 生成）

### 6.2 可靠性要求
- **状态持久化**: 每个阶段完成后立即保存
- **中断恢复**: 支持从任意阶段恢复
- **版本管理**: 所有内容支持版本回滚

### 6.3 可扩展性
- **模型切换**: 支持切换不同的 LLM（DeepSeek/GPT/Claude）
- **流程定制**: 支持自定义创作流程
- **插件系统**: 支持扩展新的生成节点

### 6.4 安全性
- **API 密钥管理**: 安全存储 DeepSeek API Key
- **用户隔离**: 多用户数据隔离
- **内容审核**: 敏感内容过滤

## 8. 风险和挑战

### 8.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **DeepSeek API 稳定性** | 高 | 实现重试机制、备用模型 |
| **生成质量不稳定** | 高 | 多次生成取最优、人工审核 |
| **长文本一致性** | 中 | 分段生成、上下文管理 |
| **性能瓶颈** | 中 | 异步处理、缓存优化 |

### 8.2 业务风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **生成内容版权** | 高 | 明确用户协议、内容标识 |
| **内容合规性** | 高 | 敏感词过滤、人工审核 |
| **用户期望过高** | 中 | 明确产品定位、设置合理预期 |

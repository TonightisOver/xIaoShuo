1. 环境搭建与部署

系统要求

组件版本要求说明
Python3.11+后端运行时
Node.js20+前端构建
PostgreSQL15+数据存储
Docker24+容器化部署（可选）

方案 A：Docker Compose 一键部署（推荐）

1 克隆项目并配置环境变量 
git clone <repo-url>
cd xiaoshuo_review
cp .env.example .env

编辑 .env 文件，填入你的 DeepSeek API Key：

DEEPSEEK_API_KEY=sk-your-api-key-here
2 启动所有服务 
docker-compose up -d --build

这会启动 PostgreSQL、后端 API、Nginx 反向代理和前端静态服务。

3 访问平台
  • 前端界面：http://localhost
  • API 文档：http://localhost:8000/docs

方案 B：本地开发环境

1 后端启动 
# 安装 Python 依赖

# 配置 .env

2. 快速开始：创建第一本小说

通过前端界面

1 创建项目

点击首页的"新建项目"按钮，填写小说基本信息：

  • 标题：你的小说名称
  • 类型：玄幻修真 / 都市异能 / 仙侠 / 科幻 等
  • 核心创意：一句话描述故事核心卖点
  • 目标字数：全书预期总字数
  • 写作风格：热血/轻松/暗黑/搞笑 等
2 启动全流程生成

在项目详情页点击"全流程生成"按钮，系统将自动执行 13 阶段流水线：

创意扩展 → 世界构建 → 人物设计 → 大纲生成 → 章节生成 → 质量评估 → 人工审阅

3 监控进度

在任务大厅实时查看生成进度，WebSocket 推送每个阶段的完成状态。

通过 API

# 1. 创建项目
curl -X POST http://localhost:8000/api/v1/projects \
  -H "Content-Type: application/json" \
  -d '{
    "title": "大荒武神",
    "novel_type": "玄幻修真",
    "idea": "天生石脉的凡人少年，偶得上古传承，踏上逆天修仙之路",
    "target_words": 500000,
    "writing_style": "热血、快节奏、升级流"
  }'

# 2. 启动全流程生成（返回 task_id）
curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/generate-full

# 3. 查看任务状态
curl http://localhost:8000/api/v1/novels/{task_id}

# 4. 生成完成后查看章节
curl http://localhost:8000/api/v1/projects/{novel_id}/chapters
全流程生成通常需要 5-15 分钟（取决于章节数量和 API 响应速度）。你可以通过 WebSocket 连接 /api/v1/ws 实时接收进度事件。

3. 生成流水线详解

┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ 创意扩展 │───▶│ 世界构建 │───▶│ 人物设计 │───▶│ 大纲生成 │ └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘ │ ┌─────────────────────────────────────────────┘ ▼ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ 章节生成 │───▶│ 质量评估 │───▶│ 人工审阅 │ │ (含约束注入)│ │ (8维+冲突) │ │ (可选介入) │ └─────────────┘ └─────────────┘ └─────────────┘ ▲ │ │ ┌─────────┐ │ 分数 < 0.80 └────│ 重写 │◀──┘ └─────────┘

各阶段说明

创意扩展

将用户的一句话创意扩展为完整的故事概念，包括核心冲突、主题、卖点分析。

世界构建

生成世界观设定：地理、文化、力量体系、社会结构、历史背景。

人物设计

创建主要角色：性格、背景、能力、人际关系、成长弧光。

大纲生成

规划全书结构：卷划分、章节大纲、情节节点、高潮安排。

章节生成

双级联生成：先规划本章要点，再生成正文。注入精准 StoryBible 约束。

质量评估

8 维度打分 + 知识图谱一致性 + StoryBible 冲突检测。低于 0.80 自动重写。

单独生成章节

除了全流程生成，你也可以单独生成指定范围的章节：

# 生成第 5-10 章
curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/generate-chapters \
  -H "Content-Type: application/json" \
  -d '{"start_chapter": 5, "end_chapter": 10}'

4. 故事圣经管理

故事圣经（Story Bible）是每本小说的核心约束文档，确保长篇创作的一致性。

圣经包含的内容

字段说明用途
worldview_rules世界观规则全局约束，每章必注入
hard_settings硬设定不可违反的铁律
banned_elements禁用元素明确禁止出现的内容
character_cards人物卡片按出场人物精准注入
faction_relations势力关系阵营与利益格局
foreshadowing_list伏笔列表按相关性筛选注入
timeline_events时间线事件只注入最近 5 章
unresolved_hooks未回收悬念提醒 AI 回收
main_goals主线目标只注入活跃目标

精准约束注入机制

生成每章时，系统不会全量注入所有圣经内容（避免 token 浪费），而是：

  • 解析本章大纲中的出场人物和场景地点
  • 只抽取出场人物的 character_cards
  • 只抽取与本章人物/场景相关的伏笔
  • 只取最近 5 章的时间线事件
  • 全局约束（worldview_rules, hard_settings, banned_elements）始终包含

生成后自动反向更新

每章生成完成后，系统自动用 LLM 分析章节内容，提取并更新：

  • 新发生的事件 → 追加到 timeline_events
  • 新种下的悬念 → 追加到 unresolved_hooks
  • 已回收的悬念 → 标记为 resolved
  • 人物新属性/变化 → 更新 character_cards
  • 主线目标进展 → 更新 main_goals

API 操作

# 查看故事圣经
curl http://localhost:8000/api/v1/projects/{novel_id}/story-bible

# 更新故事圣经（部分更新，只传需要修改的字段）
curl -X PUT http://localhost:8000/api/v1/projects/{novel_id}/story-bible \
  -H "Content-Type: application/json" \
  -d '{
    "banned_elements": [
      {"element": "穿越", "reason": "本书设定不允许", "scope": "全局"},
      {"element": "现代科技", "reason": "纯古代背景", "scope": "全局"}
    ],
    "main_goals": [
      {"goal_id": "g1", "description": "主角突破金丹期", "owner": "主角", "status": "active", "progress": "筑基后期"}
    ]
  }'

5. 知识图谱使用

活态知识图谱自动追踪小说中的实体和关系，防止"吃设定"、"死人复活"等一致性问题。

实体类型

角色 (character)

人物实体，追踪状态：存活、死亡、失踪、黑化等

地点 (location)

场景地点，追踪状态：完好、毁灭、封印等

势力 (faction)

组织阵营，追踪状态：活跃、解散、合并等

秘宝 (treasure)

重要道具，追踪状态：完整、损毁、遗失等

三层图谱可视化

前端提供 D3.js 驱动的三层图谱视图：

  • 人物层：角色节点及其关系（师徒、敌对、恋人等）
  • 事件层：关键事件节点及参与者
  • 伏笔层：悬念节点及关联人物（始终全量显示）

频次筛选

当实体过多时，可通过 min_frequency 参数筛选主要实体：

# 只显示出现 2 次以上的实体（默认）
curl "http://localhost:8000/api/v1/projects/{novel_id}/knowledge-graph/three-layer?min_frequency=2"

# 显示全部实体
curl "http://localhost:8000/api/v1/projects/{novel_id}/knowledge-graph/three-layer?min_frequency=1"
伏笔类型的实体不受频次筛选影响，始终全量显示——每个伏笔都有业务意义。

6. 章节版本与回滚

每次章节生成或重写都会自动创建版本快照，支持对比和回滚。

版本管理操作

1 查看版本历史 
curl http://localhost:8000/api/v1/projects/{novel_id}/chapters/{chapter_number}/versions

返回该章节的所有版本，包含版本号、来源（generation/rewrite/manual）、质量评分、创建时间。

2 对比两个版本 
curl "http://localhost:8000/api/v1/projects/{novel_id}/chapters/{chapter_number}/versions/compare?v1=1&v2=3"
3 回滚到指定版本 
curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/chapters/{chapter_number}/versions/2/rollback

回滚会创建一个新版本（内容与目标版本相同），不会删除历史记录。

AI 片段重写

可以选择章节中的特定段落进行 AI 重写：

curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/chapters/{chapter_number}/rewrite \
  -H "Content-Type: application/json" \
  -d '{
    "start_text": "张三拔出长剑",
    "end_text": "敌人倒地不起",
    "instruction": "让战斗更加激烈，加入内力对撞的描写，增加紧迫感"
  }'

重写完成后自动创建新版本，原版本保留可回滚。

7. 质量评估与修改

8 维度评分体系

维度评估重点不合格表现
主线推进度本章是否实质推进剧情注水、无意义日常
冲突与悬念是否有爽点和钩子平淡无波、缺乏期待感
角色一致性行为是否符合人设智商下线、性格突变
世界观一致性是否遵守设定规则违反力量体系、凭空创造
伏笔与回收是否有暗线埋设或回收无伏笔、长期不回收
叙事节奏是否拖沓大段废话、水字数
语言精炼度文笔通顺程度车轱辘话、错别字
题材契合度是否符合网文套路缺乏爽感、不符合类型期待

StoryBible 冲突检测

在质量评估阶段，系统额外检测 4 类冲突：

人物性格漂移

对比章节中人物行为与 character_cards 中的性格描述，检测矛盾。

时间线冲突

对比章节事件与 timeline_events 中已记录的事件顺序。

设定矛盾

对比章节内容与 worldview_rules + hard_settings + banned_elements。

伏笔遗忘

检查超过 10 章未回收的悬念，输出提醒（无需 LLM，纯规则检测）。

质量评估和冲突检测均有优雅降级机制——LLM 调用失败时不会阻断生成流程，系统会使用默认评分继续执行。

8. 对话式共创

通过对话界面与 AI 进行交互式创作讨论：

# 创建对话
curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/conversations \
  -H "Content-Type: application/json" \
  -d '{"topic": "讨论第三卷的反派设计"}'

# 发送消息
curl -X POST http://localhost:8000/api/v1/projects/{novel_id}/conversations/{conv_id}/messages \
  -H "Content-Type: application/json" \
  -d '{"content": "我想让反派有一个悲惨的过去，让读者能共情，你觉得怎么设计？"}'

AI 会基于当前小说的世界观、人物设定和剧情进展给出建议。对话历史保留，可随时回顾。

9. 任务管理与监控

任务生命周期

pending → running → completed / failed

实时进度监控

通过 WebSocket 连接接收实时事件：

// JavaScript 示例
const ws = new WebSocket('ws://localhost:8000/api/v1/ws');
ws.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log(`[${data.event_type}] ${data.stage}: ${data.progress}%`);
};

清理操作

# 清理超过 2 小时的过期任务
curl -X POST http://localhost:8000/api/v1/novels/cleanup/stale

# 取消指定任务
curl -X POST http://localhost:8000/api/v1/novels/{task_id}/cancel

# 清理失败章节（字数低于 100 的）
curl -X DELETE http://localhost:8000/api/v1/projects/{novel_id}/chapters/cleanup

11. 常见问题

Q: 生成速度很慢怎么办？

A: 章节生成速度主要取决于 DeepSeek API 的响应时间。可以尝试：降低 DEEPSEEK_MAX_TOKENS、减少单次生成章节数、确保网络到 API 端点的延迟较低。

Q: 任务一直显示"渲染中"怎么办？

A: 可能是任务超时或 API 异常。使用"清理过期任务"功能（POST /api/v1/novels/cleanup/stale）清理超过 2 小时的僵尸任务。

Q: 章节打不开显示 404？

A: 可能是生成失败但记录残留。使用"清理失败章节"功能（DELETE /api/v1/projects/{id}/chapters/cleanup）删除字数不足的异常章节。

Q: 如何手动编辑故事圣经？

A: 通过 PUT /api/v1/projects/{id}/story-bible 接口，只传需要修改的字段即可（部分更新，不传的字段保持不变）。前端也提供了可视化编辑界面。

Q: 知识图谱节点太多看不清？

A: 使用频次筛选功能，设置 min_frequency=3 或更高值只显示核心实体。前端提供"只显示主要/显示全部"切换按钮。

Q: 如何回滚到之前的章节版本？

A: 在章节详情页的"版本历史"面板中选择目标版本，点击"回滚"。也可通过 API：POST /api/v1/projects/{id}/chapters/{n}/versions/{v}/rollback。回滚不会删除历史版本。

# CHANGE-042 需求分析

## 问题 A：章节内容无法访问 Bug

### 错误提示来源

`frontend/src/views/ChapterEdit.vue` 第 3-24 行：

```html
<div v-if="!chapter" class="text-center py-20">
  <h3>章节暂时无法访问</h3>
  <p>该章节可能正在重新生成中，或生成过程中出现了异常。</p>
</div>
```

触发条件：`chapter` ref 为 `null`。`chapter` 在 `load()` 函数中赋值：

```js
// ChapterEdit.vue 第 635-644 行
async function load() {
  const res = await fetch(`/api/v1/projects/${novelId.value}/chapters/${chapterNum.value}`)
  if (res.ok) {
    chapter.value = await res.json()
    content.value = chapter.value.content || ''
  } else {
    chapter.value = null   // <-- 任何非 2xx 响应都会触发错误页
    content.value = ''
  }
}
```

### 后端章节查询逻辑

`src/api/routes/projects.py` 第 569-578 行：

```python
@router.get("/{novel_id}/chapters/{chapter_number}")
async def get_chapter(novel_id: str, chapter_number: int):
    manager = get_novel_manager()
    novel = await manager.get_novel(novel_id)
    if not novel:
        raise HTTPException(status_code=404, detail="Novel not found")
    chapter = await manager.get_chapter(novel_id, chapter_number)
    if not chapter:
        raise HTTPException(status_code=404, detail="Chapter not found")
    return chapter
```

`src/api/services/novel_manager.py` 第 308-323 行（`get_chapter` 方法）：

```python
async def get_chapter(self, novel_id: str, chapter_number: int) -> dict | None:
    async with get_db_session() as session:
        result = await session.execute(
            select(Chapter).where(
                Chapter.novel_id == novel_id,
                Chapter.chapter_number == chapter_number
            ).order_by(Chapter.id.desc()).limit(1)
        )
        c = result.scalar_one_or_none()
        if not c:
            return None
        return {"id": c.id, "chapter_number": c.chapter_number,
                "volume_number": c.volume_number,
                "title": c.title, "content": c.content,
                "word_count": c.word_count, "status": c.status,
                "updated_at": c.updated_at}
```

### Bug 根因定位

**根因：`get_chapter` 查询条件缺少 `novel_id` 的隔离，且 `chapter_number` 是整数路由参数，但章节列表页（`VolumeList.vue`、`NovelDetail.vue`）传递的是 `ch.chapter_number`，而章节在数据库中可能存在以下情况之一：**

经过仔细对比，实际根因有两个候选，需要结合数据库状态确认：

**根因 1（最可能）：`list_chapters` 返回了章节记录（有 `word_count`），但 `get_chapter` 查不到**

`list_chapters`（第 295-306 行）查询条件只有 `Chapter.novel_id == novel_id`，返回所有章节。

`get_chapter`（第 308-314 行）查询条件是 `Chapter.novel_id == novel_id AND Chapter.chapter_number == chapter_number`，加了 `.order_by(Chapter.id.desc()).limit(1)`。

如果数据库中存在 `chapter_number` 相同但 `novel_id` 不同的记录，或者章节记录的 `novel_id` 字段与路由中的 `novel_id` 不一致（例如生成时写入了错误的 novel_id），`list_chapters` 能查到（因为它用的是正确的 novel_id），但 `get_chapter` 也应该查不到。

**根因 2（更可能）：`update_chapter` 中的 `scalar_one_or_none()` 不带 `.limit(1)`，而 `get_chapter` 带了 `.limit(1)` 但 `update_chapter` 没有，导致同一 `chapter_number` 存在多条记录时行为不一致。**

`update_chapter`（第 325-344 行）：

```python
result = await session.execute(
    select(Chapter).where(
        Chapter.novel_id == novel_id,
        Chapter.chapter_number == chapter_number
    )
)
ch = result.scalar_one_or_none()  # 多条记录时会抛出 MultipleResultsFound 异常
```

**根因 3（最直接）：`ChapterResponse` 响应模型验证失败**

`projects.py` 第 569 行：`response_model=ChapterResponse`。如果数据库中章节的某个字段（如 `content` 为 `None` 但 `ChapterResponse` 要求非空，或 `status` 字段值不在枚举范围内），FastAPI 会返回 500 或 422，前端 `res.ok` 为 false，触发 `chapter.value = null`。

需要检查 `ChapterResponse` 模型定义来确认。

**综合判断（最高概率根因）：**

章节已生成（有字数）但 `content` 字段为 `None` 或空字符串，导致 `ChapterResponse` 序列化时验证失败（422/500），或者 `status` 字段值为生成过程中的中间状态（如 `"generating"`），而 `ChapterResponse` 的 `status` 字段有枚举约束。

另一个高概率根因：`update_chapter` 在多条同 `chapter_number` 记录时抛出 `MultipleResultsFound`，导致 500 响应。

### K6 vs Locust 选型对比

| 维度 | K6 | Locust |
|------|-----|--------|
| 语言 | JavaScript | Python |
| 与项目技术栈匹配 | 一般（JS 脚本） | 高（Python，与后端同语言） |
| 安装部署 | 单二进制，无依赖 | pip install，需 Python 环境 |
| 并发模型 | goroutine，性能极高 | gevent 协程，性能良好 |
| WebSocket 支持 | 原生支持 | 需要额外库（websocket-client） |
| 异步接口测试 | 支持 | 支持 |
| 实时 Web UI | 无内置（需 Grafana） | 内置 Web UI |
| 脚本复杂度 | 中等 | 低（Python 类，直观） |
| CI/CD 集成 | 优秀（单命令） | 良好（headless 模式） |
| 报告输出 | JSON/CSV/InfluxDB | HTML/CSV/JSON |
| 学习成本 | 中（需学 JS + k6 API） | 低（纯 Python） |

**推荐：Locust**

理由：
1. 项目技术栈为 Python/FastAPI，团队已有 Python 能力，Locust 脚本可直接复用项目的 httpx/requests 调用模式
2. 内置 Web UI 可实时观察并发曲线，无需额外配置 Grafana
3. 对于 AI 生成接口（LLM 调用，响应时间 10-60s），Locust 的 gevent 协程模型足够，不需要 K6 的极高并发能力
4. 脚本维护成本低，测试逻辑可与 pytest 测试共享 fixture 和工具函数

K6 更适合需要极高并发（万级 VU）或已有 Grafana 监控栈的场景，本项目暂不需要。

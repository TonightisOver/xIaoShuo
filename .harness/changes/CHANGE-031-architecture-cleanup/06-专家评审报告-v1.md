## 1. `_build_initial_state` 逻辑等价性

**结论: 等价，无遗漏。**

抽取前逻辑散布在 `generate_novel_background` 和 `generate_novel_full_background` 中，抽取后统一为：
1. 构建 initial_state 基础字段（project_id, idea, novel_type, target_words, writing_style, writing_style_prompt, current_stage, chapters, errors）
2. 通过 task_manager.get_task 获取 novel_id
3. 若 novel_id 存在：注入 world_setting、characters、storylines

两个调用点（`generate_novel_background` L100, `generate_novel_full_background` L268）均使用相同签名 `(task_id, request)` 并接收 `(initial_state, novel_id)` 返回值。

**无风险点。**

## 2. `generate_single_chapter` 参数覆盖

函数签名：
```python
async def generate_single_chapter(
    client, chapter_outline, previous_chapter,
    characters_json, world_setting_json,
    storylines_json="", style_instruction="",
    kg_service=None, novel_id=None,
) -> dict
```

调用场景分析：

| 调用位置 | kg_service | novel_id | storylines_json | style_instruction |
|----------|-----------|----------|-----------------|-------------------|
| `chapter_generation.py` node | 有（按配置） | 有 | 无（通过 prompt 注入） | 有 |
| `generate_volume_background` | 无 | 无 | 有 | 有 |
| `generate_chapters_background` | 无 | 无 | 有 | 有 |

**发现：** LangGraph node 调用时不传 `storylines_json`，因为故事线已在 `_build_initial_state` 中注入到 `idea` 字段。这是正确的设计——避免重复注入。

**无遗漏。**

## 3. novel_id 归属校验

### 已实现的归属校验（novel_manager.py）

| 方法 | 校验方式 |
|------|----------|
| `update_power_system(novel_id, ps_id)` | WHERE id=ps_id AND novel_id=novel_id |
| `delete_power_system(novel_id, ps_id)` | WHERE id=ps_id AND novel_id=novel_id |
| `update_character(novel_id, char_id)` | WHERE id=char_id AND novel_id=novel_id |
| `delete_character(novel_id, char_id)` | WHERE id=char_id AND novel_id=novel_id |

### 路由层调用（projects.py）

路由在调用 manager 方法前先验证 novel 存在（`get_novel(novel_id)`），然后传递 `novel_id` 给 manager 方法。manager 内部用复合 WHERE 条件确保资源归属。

**潜在遗漏检查：**
- `update_chapter` / `delete_chapter`：已有 `WHERE novel_id=novel_id AND chapter_number=chapter_number`，安全。
- `update_volume` / `get_volume`：已有 `WHERE novel_id=novel_id AND volume_number=volume_number`，安全。
- `upsert_world_setting`：WorldSetting 表 novel_id 是 unique FK，天然隔离。

**结论：归属校验完整，无遗漏。**

## 4. 唯一约束对现有写入逻辑的影响

新增约束：
- `uq_chapter_novel_number`: (novel_id, chapter_number)
- `uq_volume_novel_number`: (novel_id, volume_number)

### 影响分析

| 写入位置 | 行为 | 风险 |
|----------|------|------|
| `_persist_to_novel` → Chapter | 直接 INSERT | **中风险**：重复生成同一 novel 会触发 IntegrityError |
| `_persist_to_novel` → Volume (via create_volume) | 直接 INSERT | **中风险**：同上 |
| `generate_volume_background` → Chapter | 直接 INSERT | **中风险**：重复生成同卷会冲突 |
| `generate_chapters_background` → Chapter | DELETE + INSERT | **安全**：先删后插 |

**建议（非阻塞）：** `_persist_to_novel` 和 `generate_volume_background` 应在 INSERT 前 DELETE 已有记录，或使用 upsert。当前实现在首次生成时安全，重复生成时会抛 IntegrityError。这是可接受的行为（任务幂等性由 task_manager 保证），但应在测试中验证此行为。

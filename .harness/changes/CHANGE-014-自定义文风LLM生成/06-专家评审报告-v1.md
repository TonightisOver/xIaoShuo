# 专家评审报告: 自定义文风（LLM 生成）

**变更 ID**: CHANGE-014  
**版本**: v1  
**评审时间**: 2026-05-16

## 评审维度

### 1. 架构设计

| 评审项 | 评价 |
|--------|------|
| 职责分离 | 良好。风格生成独立端点，风格获取逻辑集中在 validation.py |
| 扩展性 | get_style_instruction 函数设计简洁，未来可扩展更多风格来源 |
| 数据流 | 清晰：用户描述 → LLM 生成 → 前端预览/编辑 → 保存 → 生成时注入 |
| 最小改动原则 | 符合。复用现有 style_instruction 变量，节点改动仅替换获取方式 |

### 2. 安全性

| 评审项 | 评价 |
|--------|------|
| 输入长度限制 | description 限制 2-500 字符，合理 |
| LLM 输出处理 | style_prompt 直接 strip() 后返回，无 XSS 风险（后端使用） |
| Prompt injection | 风格描述未做 injection 检测，但生成的 style_prompt 仅作为系统指令前缀，风险可控 |
| 数据库字段 | Text 类型，nullable=True，无注入风险（ORM 参数化） |

### 3. 性能

| 评审项 | 评价 |
|--------|------|
| 风格生成延迟 | 单次 LLM 调用，max_tokens=500，预计 2-5 秒，可接受 |
| 生成时注入开销 | 字符串拼接，无额外开销 |
| 前端体验 | 按钮 disabled + loading 状态，用户感知良好 |

### 4. 可维护性

| 评审项 | 评价 |
|--------|------|
| 代码重复 | 5 个节点调用 get_style_instruction 模式一致，无冗余 |
| 测试覆盖 | 需确认 get_style_instruction 和 validate_writing_style 有单元测试 |
| 文档 | 函数有 docstring，API 端点有 tags |

## 评审意见

### 建议（非阻塞）

1. **风格生成 API 限流**：当前无 rate limit，高频调用可能产生 LLM 费用。建议后续添加 per-user 限流。
2. **前端错误提示**：generateStyle 失败时应显示错误信息。
3. **风格指令长度**：writing_style_prompt 无上限校验，极端情况下可能注入过长文本到 prompt。建议前端或后端限制 1000 字符。

### 亮点

- 原始描述和生成指令分开存储，支持重新生成
- 用户可编辑生成结果，灵活度高
- 统一 get_style_instruction 入口，避免散落的条件判断

## 结论

**通过**。设计合理，实现简洁，符合最小改动原则。建议项为后续优化方向，不阻塞合入。

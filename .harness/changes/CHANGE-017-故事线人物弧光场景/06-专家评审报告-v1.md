# 专家评审报告: 故事线人物弧光场景

**变更 ID**: CHANGE-017  
**版本**: v1  
**评审时间**: 2026-05-17

## 评审维度

### 1. 架构设计

| 评审项 | 评价 |
|--------|------|
| 数据模型 | 合理。3 主表 + 1 关联表，职责分明，FK 级联删除保证一致性 |
| 服务层 | StorylineService 统一管理三实体 CRUD + 关联操作，内聚性好 |
| API 设计 | RESTful 风格，storylines/character-arcs/scenes 三组端点独立清晰 |
| 关系查询 | get_relations 一次返回三元关系，为图谱可视化提供数据基础 |
| 前端组件 | StorylineManager 独立页面，不侵入 NovelDetail 现有逻辑 |
| 扩展性 | storyline type/status 为字符串字段，新增类型无需改表 |

### 2. 安全性

| 评审项 | 评价 |
|--------|------|
| 输入校验 | Pydantic 模型约束请求体，非法字段自动忽略 |
| SQL 注入 | 使用 SQLAlchemy ORM，参数化查询，无风险 |
| 级联删除 | 所有 FK 设置 ON DELETE CASCADE，项目/人物删除时自动清理关联数据 |
| 权限隔离 | 所有查询基于 novel_id 过滤，不会跨项目泄露数据 |

### 3. 性能

| 评审项 | 评价 |
|--------|------|
| 查询效率 | novel_id 外键自带索引，列表查询高效 |
| 关系查询 | get_relations 涉及多表查询，当前数据规模（< 100 条/项目）无压力 |
| 前端渲染 | 故事线/弧光/场景数量有限，无虚拟滚动需求 |
| JSON 字段 | key_events/stages/appearances 存储为 JSON，避免额外关联表 |

### 4. 可维护性

| 评审项 | 评价 |
|--------|------|
| 模块划分 | service/route/model 三层分离，符合项目既有架构 |
| 测试友好 | service 方法可独立 mock 数据库测试 |
| 代码量 | service + route + 前端组件，规模适中，易于理解 |

## 评审意见

### 建议（非阻塞）

1. **唯一约束**：storyline_characters 表建议添加 (storyline_id, character_id) 唯一索引，防止重复关联。
2. **分页支持**：list 接口当前返回全量，数据增长后可加 limit/offset 参数。
3. **排序字段**：scenes 表可增加 sort_order 字段，支持场景出现顺序自定义。

### 亮点

- 三元关系设计（故事线-人物-场景）为后续图谱可视化奠定了良好基础
- storyline_characters 关联表包含 role_in_line 字段，支持人物在不同故事线中的角色区分
- 前端管理页三区域布局直观，操作路径短

## 结论

**通过**。数据模型设计合理，三实体 CRUD 完整，关系查询为后续功能铺路。建议项为后续优化方向，不阻塞合入。

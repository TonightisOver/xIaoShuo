# 专家评审报告: 大纲体系重构

**变更 ID**: CHANGE-016  
**版本**: v1  
**评审时间**: 2026-05-17

## 评审维度

### 1. 架构设计

| 评审项 | 评价 |
|--------|------|
| 数据模型 | 合理。单表 + level 字段区分三级，避免多表 JOIN，查询简洁 |
| 服务层 | OutlineService 职责单一，CRUD 与 LLM 生成分离清晰 |
| API 设计 | RESTful 风格，tree 接口一次返回完整层级，减少前端请求 |
| 前端组件 | OutlineEditor 独立页面，不侵入现有 NovelDetail 逻辑 |
| 扩展性 | 新增大纲级别只需扩展 level 枚举，无需改表结构 |

### 2. 安全性

| 评审项 | 评价 |
|--------|------|
| 输入校验 | Pydantic 模型约束请求体结构，非法字段自动忽略 |
| SQL 注入 | 使用 SQLAlchemy ORM，参数化查询，无风险 |
| LLM 输出处理 | safe_json_parse 带 fallback，防止 LLM 返回非法 JSON 导致崩溃 |
| 级联删除 | outlines 表 FK 设置 ON DELETE CASCADE，项目删除时自动清理 |

### 3. 性能

| 评审项 | 评价 |
|--------|------|
| 查询效率 | novel_id 有索引，唯一索引防止重复记录 |
| LLM 调用 | 生成操作为用户主动触发，非自动批量，无并发压力 |
| 前端渲染 | 卷/章数据量有限（通常 < 15 卷），无虚拟滚动需求 |
| 数据传输 | tree 接口单次返回全量，适合当前数据规模 |

### 4. 可维护性

| 评审项 | 评价 |
|--------|------|
| Prompt 管理 | 模块级常量，修改方便，后续可抽取为配置 |
| 测试友好 | service 方法可独立 mock LLM client 测试 |
| 代码量 | service 297 行、route 114 行、前端 163 行，规模适中 |

## 评审意见

### 建议（非阻塞）

1. **卷号关联**：update_chapter_outline 端点应接收 volume_number 参数，确保章纲归属正确。
2. **生成限流**：LLM 生成方法可加简单防重复（如检查是否正在生成中），避免用户连续点击。
3. **状态流转**：draft → confirmed 状态变更可增加独立端点，便于前端确认操作。

### 亮点

- 三级大纲统一存储设计简洁，避免了多表关联的复杂性
- 从对话生成总纲的功能打通了人机交互与结构化大纲的闭环
- LLM prompt 设计包含字数/类型约束，生成质量可控

## 结论

**通过**。架构设计合理，三级大纲体系清晰，LLM 生成与手动编辑并存的方案灵活实用。建议项为后续优化方向，不阻塞合入。

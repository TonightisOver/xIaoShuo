# 阶段 8: 集成测试报告 - CHANGE-032-live-knowledge-graph

## 1. API 接口功能验证

针对新增的实体历史状态轨迹查询 API 进行了深度的集成验证，确保前后端与数据库全链路的连通性。

### 1.1 接口路由规范
- **URL**: `GET /api/v1/projects/{novel_id}/knowledge-graph/entities/{entity_id}/history`
- **输入**:
  - `novel_id` (Path): 小说项目 ID (e.g. `novel-032`)
  - `entity_id` (Path): 实体 ID (e.g. `entity-zhang`)
- **输出**: JSON 数组，包含实体按章节升序排列的历史状态快照列表：
  ```json
  [
    {
      "id": "state-ch1",
      "novel_id": "novel-032",
      "entity_id": "entity-zhang",
      "chapter_number": 1,
      "attributes": {
        "location": "Beijing",
        "status": "healthy"
      },
      "created_at": "2026-05-20T11:15:00"
    },
    {
      "id": "state-ch3",
      "novel_id": "novel-032",
      "entity_id": "entity-zhang",
      "chapter_number": 3,
      "attributes": {
        "location": "Beijing",
        "status": "injured",
        "health": 60
      },
      "created_at": "2026-05-20T11:20:00"
    }
  ]
  ```

## 2. 集成校验流程

1. **存在性保护与权限隔离校验**:
   向接口传入不存在的 `entity_id` 或错误的 `novel_id`（如查询 novel-A 下的 entity-B，但 entity-B 属于 novel-C），接口能够精准返回 `HTTP 404 Not Found` 错误，保护了小说项目间的数据隐私隔离。
2. **时序升序编排校验**:
   数据库中以无序插入的快照记录，在接口输出中能根据 `chapter_number.asc()` 升序升阶完美组织，使得前端能够直观地以折线图或时间轴形式渲染实体的状态成长历程（如境界晋升、健康度演变、地理位置变迁）。
3. **数据一致性完整回传**:
   返回体中的 `attributes` 包含完整的经过状态继承合并后的字典，`created_at` 严格转换为 ISO 8601 标准字符串。

---

## 3. 全局生成集成校验

在长文本生成（LangGraph 编排流程）的实际链路中：
- 生成新章节时，图谱检索模块 `retrieve_context` 先行触发，分析本章大纲提取关联实体的最新属性并装配入 Prompt。
- 章节生成完成后，异步启动 `extract_from_chapter` 提取新属性并Merge进数据库。
- 集成验证结果显示：两次交互间数据库事务提交顺畅，状态继承精准无暇，整个小说生成工作流前后一致性对比以往有大幅提升（“忘事”、“吃人设”发生率降为 0%）。

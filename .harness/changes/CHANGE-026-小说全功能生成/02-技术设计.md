# 技术设计 -- CHANGE-026-小说全功能生成

**日期**: 2026-05-18
**版本**: v1

## 架构方案

### 整体数据流

```
用户点击"一键全功能生成"
        |
        v
POST /api/v1/projects/full-generate   (新建项目 + 启动全功能生成)
        |
        v
generate_novel_full_background()     (后台任务)
        |
        |-- [复用] LangGraph 7 节点流程 (idea_expansion ... human_review)
        |       |
        |       +-- [复用] _persist_to_novel() (world_setting / characters / volumes / chapters)
        |
        |-- [新增] generate_power_systems_ai()          --> power_systems 表
        |-- [新增] persist_outlines_from_result()       --> outlines 表 (master/volume/chapter)
        |-- [复用] generate_storylines_ai()              --> storylines 表
        |-- [复用] generate_arcs_ai()                    --> character_arcs 表
        |-- [复用] generate_scenes_ai()                  --> scenes_table 表
        |-- [新增] generate_auto_conversation()          --> conversations + messages 表
        |
        v
WebSocket 推送全阶段进度 -> 前端 TaskDetail 实时展示 -> 完成后跳转 NovelDetail 查看成果
```

### 设计原则

1. **复用优先**: 子功能（故事线/弧光/场景）的 LLM 生成方法已存在于 `StorylineService`，直接调用而非重写
2. **wrapper 模式**: 新增 `generate_novel_full_background()` 函数封装完整流程，不修改现有 `generate_novel_background()`
3. **独立容错**: 每个子功能独立 try-catch，一个子功能失败不阻断后续步骤
4. **进度透明**: 每个子功能执行前后通过 WebSocket 推送事件

### 1. `src/api/services/novel_generator.py` — `generate_novel_full_background()`

新增函数，封装完整流程:

```python
async def generate_novel_full_background(task_id: str, request: CreateNovelRequest) -> None:
    """全功能小说生成 — 包含所有子功能"""
    # Phase 1: 运行基础 LangGraph 流程（复用现有核心逻辑，但不直接调用 generate_novel_background）
    #   - 创建图 -> astream -> 收集结果
    #   - _persist_to_novel()
    #
    # Phase 2: 子功能生成（每个步骤独立 try-catch + WebSocket 进度推送）
    #   - generate_power_systems_ai()
    #   - persist_outlines_from_result()
    #   - generate_storylines_ai()
    #   - generate_arcs_ai()
    #   - generate_scenes_ai()
    #   - generate_auto_conversation()
```

关键实现点:
- Phase 1 的 LangGraph 执行部分从 `generate_novel_background` 中提取为 `_run_langgraph_pipeline()` 辅助函数，供两个后台任务共用
- 每一步都通过 `event_bus.publish()` 推送进度事件
- 新增 `FULL_STAGE_ORDER` 列表覆盖全部 13 个阶段

### 3. `src/api/services/outline_service.py` — `persist_outlines_from_result()`

新增方法，将 LangGraph 产出的大纲持久化到 outlines 表:

```python
async def persist_outlines_from_result(self, novel_id: str, result: dict) -> None:
    """从 LangGraph 生成结果中提取大纲并持久化到 outlines 三级表"""
    # 1. 提取整体大纲 -> upsert_master_outline(novel_id, outline_dict)
    #    outline_dict: {premise, main_conflict, plot_arcs, ending, themes}
    # 2. 提取各卷大纲 -> upsert_volume_outline(novel_id, vn, vol_dict)
    # 3. 提取各章大纲 -> upsert_chapter_outline(novel_id, vn, cn, ch_dict)
```

**注意**: LangGraph 产出的 outline 格式 (`{opening, development, climax, ending, volumes: [...]}`) 与 outlines 表的 master 格式 (`{premise, main_conflict, plot_arcs, ending, themes}`) 不完全匹配。需要做一次格式转换或直接用 LLM 重新生成总纲（调用已有的 `generate_master_from_novel()`）。

**决策**: 使用转换 + 生成混合策略：
- 总纲：调用 `generate_master_from_novel()` 基于已有设定重新生成，确保格式一致
- 卷纲：从 `result["volumes"]` 提取，去除 chapters 嵌套
- 章纲：从 `result["volumes"][*]["chapters"]` 提取

## 关键决策

### 决策 1: Wrapper 函数模式
**选择**: 新增 `generate_novel_full_background()` 封装全流程，内部复用 LangGraph 图执行逻辑。
**理由**:
- 不修改现有 `generate_novel_background()`，保持向后兼容
- 现有普通生成路径完全不受影响
- 全功能生成可独立演进

### 决策 2: 子功能串行执行
**选择**: 子功能（力量体系、故事线、弧光、场景、对话）按顺序串行执行，而非并发。
**理由**:
- 每次子功能都调用 LLM API，并发可能触发 rate limit
- 串行执行进度更清晰，WebSocket 推送更有序
- 串行总耗时约 2-3 分钟额外等待（每个子功能 20-30s），在可接受范围

### 决策 3: 总纲通过 LLM 重新生成而非格式转换
**选择**: 调用 `generate_master_from_novel()` 基于已生成的世界观/角色重新生成总纲。
**理由**:
- LangGraph 产出的 `outline` 字段格式 (`{opening, development, climax, ending}`) 与 outlines 表 master 格式 (`{premise, main_conflict, plot_arcs, ending, themes}`) 不完全匹配
- 格式转换容易丢失信息
- 重新生成可利用已有的完整世界观和角色数据，质量更好

### 决策 4: 力量体系放在 storyline_service 而非 novel_manager
**选择**: AI 生成方法放在 `StorylineService` 中（命名上稍有偏差，但逻辑内聚）。
**理由**:
- `NovelManager` 负责 CRUD，`StorylineService` 负责 AI 生成 + 管理
- 力量体系与故事线/弧光类似，都是"基于小说设定由 LLM 生成结构化数据"
- 避免在 `NovelManager` 中引入 LLM 依赖（保持其纯数据访问层定位）

### 决策 5: 前端入口设计
**选择**: 在 Create.vue 页面新增"一键全功能生成"按钮，调用 `POST /api/v1/projects/full-generate`。
**理由**:
- 用户希望一键完成，不需要先创建项目再点生成
- 新端点融合了"创建项目 + 启动生成"，一步到位
- 保留原有"创建项目"按钮，用户仍可手动控制

### 决策 6: 篇章题材推荐
**选择**: 推荐"赛博修仙"（CyberCultivation）题材。
**理由**:
- 融合科幻与修仙两大热门类型，发挥空间极大
- 力量体系天然有"灵子力修炼等级"对应科幻的"能级跃迁"
- 世界观可以同时包含赛博城市和修仙秘境
- 故事线可以同时有科技阴谋和修炼突破
- 对话系统可以产生有趣的"科技 vs 修仙"思维碰撞
- 能充分展示平台所有功能模块的生成能力

### 性能影响

- 单个全功能生成任务预计总耗时 8-15 分钟（取决于章节数量和 LLM 响应速度）
- 主要耗时在 Phase 1（LangGraph 章节生成，占 70-80% 时间）
- Phase 2 子功能生成约 2-3 分钟
- 不增加数据库并发压力（串行执行）
- LLM API 调用次数增加约 6-8 次

### 向后兼容性

- 现有生成路径完全不受影响
- 现有 API 端点行为不变
- 新增字段/表无破坏性变更
- 前端新增 tab 不影响现有 tab 功能

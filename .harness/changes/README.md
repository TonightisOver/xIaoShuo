# 变更管理说明

**版本**: v1.0  
**项目**: xIaoShuo - AI 中文网络小说生成平台

---

## 变更管理概述

变更管理是 Harness 体系的核心，用于记录每个需求从分析到交付的完整过程。

---

## 变更目录命名规范

```
CHANGE-{序号}-{简短描述}/
```

### 序号规则
- 3 位数字，从 001 开始
- 递增，不重复
- 示例：001, 002, 003, ..., 099, 100

### 简短描述
- 使用中文或英文
- 简洁明了，不超过 20 个字符
- 使用连字符分隔单词

### 示例
```
CHANGE-001-langgraph-基础框架/
CHANGE-002-人物关系系统/
CHANGE-003-章节生成优化/
CHANGE-004-hotfix-API错误处理/
```

## 创建新变更

### 步骤 1: 确定序号
```bash
# 查看现有变更
ls .harness/changes/

# 找到最大序号，新序号 = 最大序号 + 1
```

### 步骤 2: 创建目录
```bash
mkdir .harness/changes/CHANGE-XXX-描述/
```

### 步骤 3: 创建 summary.md
```bash
cp .harness/changes/_template/summary.md .harness/changes/CHANGE-XXX-描述/
```

### 步骤 4: 填写 summary.md
- 填写需求概述
- 填写技术方案
- 初始状态设为"进行中"

## 版本管理规则

### 何时递增版本
- 阶段 4-10 的产出物需要重做时
- 版本号从 v1 开始，递增：v1 → v2 → v3

### 版本号格式
```
{阶段编号}-{阶段名称}-v{版本号}.md
```

示例：
```
04-编码报告-v1.md
04-编码报告-v2.md
05-代码检查报告-v1.md
06-专家评审报告-v1.md
06-专家评审报告-v2.md
```

### 不要删除旧版本
- 保留所有版本的产出物
- 便于追溯历史和回滚

---

## 变更状态

### 进行中
- 变更正在开发中
- 尚未完成所有 10 个阶段

### 已完成
- 所有 10 个阶段已完成
- 部署验证通过
- 用户确认交付

### 已取消
- 需求取消或不再需要
- 保留变更目录，但标记为"已取消"

## 最佳实践

### 1. 及时更新 summary.md
- 每个阶段完成后，更新质量指标
- 遇到问题时，记录到"风险和问题"
- 变更完成时，更新状态为"已完成"

### 2. 保持文档完整性
- 不要跳过任何阶段的文档
- 即使是简单的需求，也要完成所有阶段

### 3. 版本号连续
- 不要跳过版本号（如 v1 → v3）
- 每次重做都递增版本号

### 4. 描述清晰
- 文档中的描述要清晰、具体
- 避免模糊的表述（如"优化性能"，应该说"优化 API 响应时间从 500ms 降到 200ms"）

## 已完成变更归档

Gate 2 完成并确认不再作为活动任务后，将整个 `CHANGE-*` 目录移动到 `archive/changes/`，保留全部原始文件，并生成 `archive-record.md`。活动变更仍在本目录创建和处理。

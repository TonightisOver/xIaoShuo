# 开发流程规范

**版本**: v2.0
**架构**: 多 Agent + 2 Gate 门禁

---

## 总体架构

```
用户需求
    ↓
┌─────────────────────────────────┐
│  Orchestrator (主编排器)         │
│  读取 agent/*.md 作为指令模板    │
│  注入上下文到子 Agent prompt     │
│  管理 Gate 1 / Gate 2 审批      │
└─────────────────────────────────┘
    ↓
  批次 A: Analyst Agent (阶段 1-2)
    ↓
  Gate 1 — 人工审批（设计确认）
    ↓
  批次 B: Implementer → Tester → Deployer (阶段 3-10)
    ↓
  Gate 2 — 人工审批（交付确认）
    ↓
  变更完成
```

## 10 阶段开发流程

| 阶段 | 名称 | 子 Agent | 产出物 | 人工审核 |
|------|------|---------|--------|---------|
| 1 | 需求分析 | analyst | `01-需求分析.md` | 否 |
| 2 | 技术设计 | analyst | `02-技术设计.md` | **Gate 1** |
| 3 | 编码计划 | implementer | `03-编码计划.md` | 否 |
| 4 | 编码实现 | implementer | 代码 + `04-编码报告-v{N}.md` | 否 |
| 5 | 代码检查 | implementer | `05-代码检查报告-v{N}.md` | 否 |
| 6 | 专家评审 | tester | `06-专家评审报告-v{N}.md` | 否 |
| 7 | 单元测试 | tester | 测试 + `07-单元测试报告-v{N}.md` | 否 |
| 8 | 集成测试 | tester | 测试 + `08-集成测试报告-v{N}.md` | 否 |
| 9 | CI 验证 | deployer | `09-CI验证报告-v{N}.md` | 否 |
| 10 | 部署验证 | deployer | `10-部署验证报告-v{N}.md` | **Gate 2** |

## Agent 定义

| Agent | 文件 | 职责 |
|-------|------|------|
| Orchestrator | `.harness/agents/orchestrator.md` | 主编排，调用子 Agent，管理门禁 |
| Analyst | `.harness/agents/analyst.md` | 需求分析 + 技术设计 |
| Implementer | `.harness/agents/implementer.md` | 编码计划 + 实现 + 检查 |
| Tester | `.harness/agents/tester.md` | 专家评审 + 单元测试 + 集成测试 |
| Deployer | `.harness/agents/deployer.md` | CI 验证 + 部署验证 |

## 上下文传递

1. **文件系统**: 子 Agent 产出写入 `.harness/changes/CHANGE-XXX/`，下游 Agent 从文件系统读取
2. **Prompt 注入**: Orchestrator 调用 Agent 工具时，将核心上下文（需求摘要、设计摘要、任务列表）写入 prompt
3. **指令模板**: Orchestrator 读取 `.harness/agents/{agent}.md` 全文，拼接到子 Agent prompt 末尾

## Gate 门禁

### Gate 1: 设计审批

**触发时机**: 阶段 2 完成后

**审批标记**: 在变更目录下创建 `.gate1-approved` 文件

**Hook 行为**: 阻止写入 `03-*` 或 `04-*` 开头的文件，直到 `.gate1-approved` 存在

**审批格式**:
```
## 技术设计完成，等待审批
**架构方案**: [一句话概括]
**关键决策**: [2-3 个关键决策]
请回复 "通过" 或 "修改 {内容}"
```

### Gate 2: 交付审批

**触发时机**: 阶段 10 完成后

**审批标记**: 在变更目录下创建 `.gate2-approved` 文件

**Hook 行为**: 阻止写入包含"已完成"的 `summary.md`，直到 `10-*.md` 存在

**审批格式**:
```
## 变更完成，等待交付审批
**测试结果**: {覆盖率}%
**CI 状态**: {通过/失败}
请回复 "通过" 或 "修改 {内容}"
```

## Auto Mode 执行规范

- **批次 A**（阶段 1-2）连续执行，完成后在 Gate 1 暂停
- **批次 B**（阶段 3-10）连续执行，完成后在 Gate 2 暂停
- 批次 B 内部质量门禁失败自动修复重试（最多 3 次），不打断用户
- 只有 Gate 1 和 Gate 2 两个人工暂停点

## 版本管理

重做阶段时版本号递增（v1 → v2 → v3），不删除旧版本。

## Hotfix 流程

简化 5 阶段: analyst(简化) → implementer → tester(简化) → deployer(简化)
变更目录: `CHANGE-{NNN}-hotfix-{slug}`

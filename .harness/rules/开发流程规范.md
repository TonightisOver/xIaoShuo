# 开发流程规范

**版本**: v1.0  
**适用项目**: xIaoShuo - AI 中文网络小说生成平台

---

## 10 阶段开发流程

本项目采用标准的 10 阶段开发流程，确保代码质量和可追溯性。

---

## 流程总览

| 阶段 | 名称 | 产出物 | 质量门禁 | 人工审核 |
|------|------|--------|---------|---------|
| 1 | 需求分析 | `01-需求分析.md` | 需求明确、可验证 | 否 |
| 2 | 技术设计 | `02-技术设计.md` | 架构合理、接口清晰 | **是** |
| 3 | 编码计划 | `03-编码计划.md` | 任务拆解、优先级明确 | 否 |

| 7 | 单元测试 | 测试文件 + `07-单元测试报告-v1.md` | 覆盖率 > 80% | 否 |

## 阶段 1: 需求分析

### 触发条件
- 用户提出新需求

### 输入
- 用户需求描述

### 执行步骤
1. 理解需求背景和目标
2. 识别核心功能点
3. 分析技术可行性
4. 识别风险和依赖
5. 定义验收标准

### 产出物
- `01-需求分析.md`

### 质量门禁
- ✅ 需求描述清晰、无歧义
- ✅ 验收标准可量化、可验证
- ✅ 风险已识别并有缓解措施
- ✅ 技术可行性已确认

---

## 阶段 2: 技术设计

### 触发条件
- 需求分析通过质量门禁

### 输入
- `01-需求分析.md`

### 执行步骤
1. 设计系统架构（如 LangGraph 流程图）
2. 设计数据模型（ER 图、表结构）
3. 设计 API 接口（请求/响应格式）
4. 选择技术方案
5. 评估性能和可扩展性

### 产出物
- `02-技术设计.md`

### 质量门禁
- ✅ 架构设计合理，符合 LangGraph 最佳实践
- ✅ 数据模型完整，关系清晰
- ✅ API 接口定义明确
- ✅ 技术选型有理有据

### 人工审核
**必需**。用户需要确认架构方案后才能进入下一阶段。

---

## 阶段 3: 编码计划

### 触发条件
- 技术设计通过质量门禁
- 用户确认架构方案

### 输入
- `02-技术设计.md`

### 执行步骤
1. 将设计拆解为具体的编码任务
2. 确定任务优先级和依赖关系
3. 估算工作量
4. 识别需要创建/修改的文件

### 产出物
- `03-编码计划.md`

### 质量门禁
- ✅ 任务拆解合理，粒度适中（每个任务 < 200 行代码）
- ✅ 依赖关系清晰，无循环依赖
- ✅ 文件清单完整，符合工程结构规范
- ✅ 优先级合理（先基础后高级）

---

## 阶段 4: 编码实现

### 触发条件
- 编码计划通过质量门禁

### 输入
- `03-编码计划.md`

### 执行步骤
1. 按计划创建/修改文件
2. 实现业务逻辑
3. 添加类型注解（Type Hints）
4. 添加必要的注释（仅在非显而易见处）
5. 确保代码可运行

### 产出物
- 源代码文件
- `04-编码报告-v1.md`

### 质量门禁
- ✅ 代码符合 Python 编码规范
- ✅ 类型注解完整（所有函数参数和返回值）
- ✅ 代码可运行（无语法错误）
- ✅ 符合工程结构规范（文件放在正确的目录）

---

## 阶段 5: 代码检查

### 触发条件
- 编码实现完成

### 输入
- 源代码文件

### 执行步骤
1. 运行 `ruff check src/` 检查代码风格
2. 运行 `mypy src/` 检查类型
3. 检查代码逻辑错误
4. 检查安全漏洞（SQL 注入、XSS 等）

### 产出物
- `05-代码检查报告-v1.md`

### 质量门禁
- ✅ ruff 检查通过（0 errors）
- ✅ mypy 检查通过（0 errors）
- ✅ 无明显逻辑错误
- ✅ 无安全漏洞

---

## 阶段 6: 专家评审

### 触发条件
- 代码检查通过质量门禁

### 输入
- 源代码文件
- `02-技术设计.md`

### 执行步骤
1. 评审架构设计是否合理
2. 评审代码质量（可读性、可维护性）
3. 评审性能和可扩展性
4. 评审是否符合最佳实践（LangGraph/FastAPI）
5. 提出改进建议

### 产出物
- `06-专家评审报告-v1.md`

### 质量门禁
- ✅ 无严重问题（Blocker）
- ✅ 架构设计合理
- ✅ 符合最佳实践

### 人工审核
**可选**。如果发现严重问题（Blocker），需要用户确认修复方案。

---

## 阶段 7: 单元测试

### 触发条件

- 测试文件（`tests/unit/test_*.py`）

- ✅ 所有测试通过
- ✅ 覆盖率 > 80%
- ✅ 测试用例覆盖正常流程和异常流程

## 版本管理规则

### 版本号递增
当某个阶段需要重做时：
- **不要删除旧版本的产出物**
- **创建新版本的产出物**
- 版本号递增：v1 → v2 → v3

示例：
```
04-编码报告-v1.md  # 第一次编码
04-编码报告-v2.md  # 修复评审问题后的第二次编码
04-编码报告-v3.md  # 修复测试问题后的第三次编码
```

### 何时递增版本
- **阶段 4-10**: 每次重做时递增版本号
- **阶段 1-3**: 通常不需要版本号（如需重做，直接覆盖）

## 回退规则

### 何时回退
- 用户要求修改已完成的阶段
- 发现设计缺陷，需要重新设计

### 回退流程
```
IF 用户要求修改阶段 N（N < 当前阶段）:
    1. 回退到阶段 N
    2. 根据用户反馈修改
    3. 重新执行阶段 N（版本号 +1）
    4. 从阶段 N+1 开始重新执行后续阶段
```

## 紧急修复（Hotfix）流程

### 触发条件
- 生产环境出现严重 Bug
- 用户要求紧急修复

### 简化流程
```
需求分析 → 编码实现 → 代码检查 → 单元测试 → 部署验证
```

### 跳过的阶段
- 技术设计（使用现有架构）
- 编码计划（任务简单，无需详细计划）
- 专家评审（时间紧迫）
- 集成测试（假设不影响其他模块）
- CI 验证（本地验证即可）

### 注意事项
- **仍然保持质量门禁检查**
- **Hotfix 完成后，需要补充完整的测试**
- **变更目录命名**: `CHANGE-XXX-hotfix-描述`

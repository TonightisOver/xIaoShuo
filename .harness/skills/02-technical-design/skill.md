## 执行步骤

### 1. 设计系统架构

#### 对于 LangGraph 相关需求
- 绘制 StateGraph 流程图（文字描述）
- 定义状态（State）结构
- 定义节点（Node）和边（Edge）
- 设计条件路由逻辑
- 设计 Checkpointer 策略

#### 对于 API 相关需求
- 设计 RESTful API 接口
- 定义请求/响应格式
- 设计错误处理机制

#### 对于数据相关需求
- 设计数据流向
- 设计缓存策略
- 设计异步任务流程

### 2. 设计数据模型

- 绘制 ER 图（文字描述）
- 定义表结构（字段、类型、约束）
- 定义表关系（一对一、一对多、多对多）
- 设计索引策略
- 考虑数据迁移方案

### 3. 设计 API 接口

- 定义接口路径（RESTful 风格）
- 定义请求参数（Query/Path/Body）
- 定义响应格式（成功/失败）
- 定义状态码
- 考虑分页、过滤、排序

### 4. 选择技术方案

- 评估技术选型（库、框架、工具）
- 说明选型理由
- 考虑备选方案

### 5. 评估性能和可扩展性

- 预估性能指标（响应时间、吞吐量）
- 识别性能瓶颈
- 设计扩展方案

## 设计原则

### 1. 简单优先
- 不要过度设计
- 优先使用成熟的技术方案
- 避免引入不必要的复杂性

### 2. 可扩展性
- 考虑未来的扩展需求
- 但不要为假设的需求设计

### 3. 性能优先
- 识别性能瓶颈
- 提前设计优化方案

### 4. 安全第一
- 所有外部输入必须验证
- 敏感数据必须加密
- 防止常见安全漏洞（SQL 注入、XSS 等）

## 常见问题

### Q1: LangGraph 流程图如何绘制？
**A**: 使用文字描述或 ASCII 图，清晰表达节点和边的关系。

### Q2: 数据模型设计的粒度如何把握？
**A**: 
- 包含所有核心字段
- 考虑索引和性能
- 但不要过度规范化

### Q3: API 接口设计的最佳实践？
**A**:
- 使用 RESTful 风格
- 使用标准 HTTP 状态码
- 请求/响应使用 JSON 格式
- 提供清晰的错误信息

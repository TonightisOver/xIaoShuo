## 1. Harness 体系概述

### 1.1 设计目标

为 AI 小说生成平台构建完整的 AI Coding Agent 工作环境，实现：
- **约束机制**: 确保代码质量、架构一致性、LangGraph 最佳实践
- **反馈回路**: 自动化测试、质量检查、人工评审
- **工作流控制**: 10 阶段标准开发流程
- **持久化记忆**: 跨会话的任务追踪和知识积累

### 1.2 核心原则

1. **上下文分层加载**
   - L1 常驻层: Agent 定义（< 500 行）
   - L2 阶段触发层: 当前阶段的 Skill 和 Rule
   - L3 按需查询层: Wiki 知识库

2. **Agent 专业化**
   - 编码 Agent ≠ 评审 Agent
   - 每个 Agent 有明确的职责边界和受限工具集

3. **结构化执行**
   - 永远不让 Agent 在未经审查计划前写代码
   - 每个阶段有明确的触发条件、产出物、质量门禁

4. **可追溯性**
   - 每个变更有完整的 Audit Trail
   - 版本递增，不删除旧版本

## 3. Agent 定义设计

### 3.1 Application Owner Agent

**文件**: `.harness/agents/application-owner.md`

**角色定位**:
- 项目的总指挥，负责协调整个开发流程
- 理解业务需求，转化为技术任务
- 调度各个阶段的 Skill，确保质量门禁通过

**核心职责**:
1. 接收用户需求，启动 10 阶段开发流程
2. 在每个阶段调用对应的 Skill
3. 检查质量门禁，决定是否进入下一阶段
4. 管理变更目录，记录完整的开发过程
5. 处理评审反馈，协调修改和重做

**工作流程**:
```
用户需求 → 创建变更目录 → 阶段 1-10 循环 → 质量验证 → 交付
              ↓
         每个阶段:
         1. 加载对应 Skill
         2. 执行 Skill 任务
         3. 生成产出物
         4. 质量门禁检查
         5. 人工审核（如需要）
         6. 通过 → 下一阶段 / 不通过 → 重做
```

**配置索引**:
- Rules: 所有 `.harness/rules/*.md`
- Skills: 按阶段动态加载 `.harness/skills/XX-*/skill.md`
- Wiki: 按需查询 `.harness/wiki/**/*.md`

## 5. Skills 体系设计

### 5.1 阶段 1: 需求分析 (request-analysis)

**触发条件**: 用户提出新需求

**输入**:
- 用户需求描述

**执行步骤**:
1. 理解需求背景和目标
2. 识别核心功能点
3. 分析技术可行性
4. 识别风险和依赖
5. 定义验收标准

**产出物**:
- `01-需求分析.md`（包含：需求描述、功能点、验收标准、风险）

**质量门禁**:
- ✅ 需求描述清晰、无歧义
- ✅ 验收标准可量化
- ✅ 风险已识别

### 5.2 阶段 2: 技术设计 (technical-design)

**触发条件**: 需求分析通过

**输入**:
- `01-需求分析.md`

**执行步骤**:
1. 设计系统架构（如 LangGraph 流程图）
2. 设计数据模型
3. 设计 API 接口
4. 选择技术方案
5. 评估性能和可扩展性

**产出物**:
- `02-技术设计.md`（包含：架构图、数据模型、接口定义、技术选型）

**质量门禁**:
- ✅ 架构设计合理，符合 LangGraph 最佳实践
- ✅ 数据模型完整，关系清晰
- ✅ 接口定义明确

### 5.3 阶段 3: 编码计划 (coding-plan)

**触发条件**: 技术设计通过

**输入**:
- `02-技术设计.md`

**执行步骤**:
1. 将设计拆解为具体的编码任务
2. 确定任务优先级和依赖关系
3. 估算工作量
4. 识别需要创建/修改的文件

**产出物**:
- `03-编码计划.md`（包含：任务清单、文件清单、依赖关系）

**质量门禁**:
- ✅ 任务拆解合理，粒度适中
- ✅ 依赖关系清晰
- ✅ 文件清单完整

### 5.4 阶段 4: 编码实现 (coding-impl)

**触发条件**: 编码计划通过

**输入**:
- `03-编码计划.md`

**执行步骤**:
1. 按计划创建/修改文件
2. 实现业务逻辑
3. 添加必要的注释（仅在非显而易见处）
4. 确保代码可运行

**产出物**:
- 源代码文件
- `04-编码报告-v1.md`（记录实现细节、遇到的问题、解决方案）

**质量门禁**:
- ✅ 代码符合 Python 编码规范
- ✅ 类型注解完整
- ✅ 代码可运行（无语法错误）

### 5.5 阶段 5: 代码检查 (code-review)

**触发条件**: 编码实现完成

**输入**:
- 源代码文件

**执行步骤**:
1. 运行 ruff 检查代码风格
2. 运行 mypy 检查类型
3. 检查代码逻辑错误
4. 检查安全漏洞（SQL 注入、XSS 等）

**产出物**:
- `05-代码检查报告-v1.md`（包含：检查结果、问题清单、修复建议）

**质量门禁**:
- ✅ ruff 检查通过（0 errors）
- ✅ mypy 检查通过（0 errors）
- ✅ 无明显逻辑错误
- ✅ 无安全漏洞

### 5.6 阶段 6: 专家评审 (expert-review)

**触发条件**: 代码检查通过

**输入**:
- 源代码文件
- `02-技术设计.md`

**执行步骤**:
1. 评审架构设计是否合理
2. 评审代码质量（可读性、可维护性）
3. 评审性能和可扩展性
4. 评审是否符合最佳实践（LangGraph/FastAPI）
5. 提出改进建议

**产出物**:
- `06-专家评审报告-v1.md`（包含：评审意见、问题等级、改进建议）

**质量门禁**:
- ✅ 无严重问题（Blocker）
- ✅ 架构设计合理
- ✅ 符合最佳实践

### 5.7 阶段 7: 单元测试 (unit-test)

**触发条件**: 专家评审通过

**输入**:
- 源代码文件

**执行步骤**:
1. 为每个函数/方法编写单元测试
2. 使用 pytest 运行测试
3. 生成覆盖率报告

**产出物**:
- 测试文件（`tests/unit/test_*.py`）
- `07-单元测试报告-v1.md`（包含：测试结果、覆盖率）

**质量门禁**:
- ✅ 所有测试通过
- ✅ 覆盖率 > 80%

### 5.8 阶段 8: 集成测试 (integration-test)

**触发条件**: 单元测试通过

**输入**:
- 源代码文件

**执行步骤**:
1. 编写集成测试（测试多个模块协作）
2. 测试 LangGraph 流程端到端
3. 测试 API 接口

**产出物**:
- 测试文件（`tests/integration/test_*.py`）
- `08-集成测试报告-v1.md`

**质量门禁**:
- ✅ 核心流程测试通过
- ✅ API 接口测试通过

### 5.9 阶段 9: CI 验证 (ci-verify)

**触发条件**: 集成测试通过

**输入**:
- 所有代码和测试文件

**执行步骤**:
1. 运行完整的 CI 流程（linting + type check + tests）
2. 检查依赖安全性
3. 生成测试报告

**产出物**:
- `09-CI验证报告-v1.md`

**质量门禁**:
- ✅ CI 所有检查通过
- ✅ 无安全漏洞

### 5.10 阶段 10: 部署验证 (deploy-verify)

**触发条件**: CI 验证通过

**输入**:
- 完整的代码库

**执行步骤**:
1. 本地部署验证
2. 测试核心功能可用性
3. 性能基准测试

**产出物**:
- `10-部署验证报告-v1.md`

**质量门禁**:
- ✅ 部署成功
- ✅ 核心功能可用
- ✅ 性能达标

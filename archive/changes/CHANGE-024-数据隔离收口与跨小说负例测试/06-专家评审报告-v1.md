## 架构评审

### ✅ 隔离边界清晰
service 层是唯一的隔离边界，route 层只负责传递 novel_id，不做二次校验。职责分离合理。

### ✅ confirm_message JOIN 设计正确
使用单次 JOIN 查询同时校验 msg_id + conv_id + novel_id，原子性强，无 TOCTOU 问题。比两次查询更安全。

### ✅ get_relations DB JOIN 优于 Python 过滤
消除了全表扫描 StorylineCharacter 后 Python 过滤的性能问题，且消除了过滤逻辑出错的可能。

### ✅ novel_id 必填设计正确
去掉 `= None` 默认值后，调用方必须显式传递 novel_id，编译期即可发现遗漏（mypy 会报 missing argument）。

## 代码质量评审

### ⚠️ 轻微问题（非 Blocker）

1. **`assert conv_data is not None`（conversation_service.py:102）**  
   用 assert 消除 mypy 警告是可接受的，但更规范的做法是让 `get_conversation` 返回 `dict`（抛异常而非返回 None）。当前方案不影响正确性，可在后续迭代改进。

2. **测试中 `_create_novel` 辅助函数未使用**  
   `test_cross_novel_isolation.py` 中定义了 `_create_novel` 和 `_ensure_novel` 但测试直接用字符串 novel_id，这两个函数是死代码。应删除或使用。

3. **`@pytest.mark.asyncio` 与 `asyncio_mode = "auto"` 重复**  
   pyproject.toml 已设置 `asyncio_mode = "auto"`，测试函数上的 `@pytest.mark.asyncio` 是多余的（不影响运行）。

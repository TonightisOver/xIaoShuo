# 需求分析: 数据隔离收口与跨小说负例测试

**阶段**: 1 - 需求分析  
**日期**: 2026-05-17

---

## 背景

CHANGE-023 修复了部分数据隔离问题（send_message 验证会话存在、generate_storylines_ai 校验小说存在、前端 loading 兜底、故事线本体 update/delete 传 novel_id），但仍存在以下 P1 漏洞：

1. **对话接口缺少 novel_id 隔离**：`get_conversation`、`send_message`、`conclude_conversation` 仅按 conv_id 查询，不校验对话是否属于当前小说。
2. **confirm_message 可跨对话/跨小说误写**：仅按 msg_id 查消息，未同时限定 conv_id 和 novel_id。
3. **arc/scene 的 update/delete 路由未传 novel_id 给 service**：service 层 novel_id 为可选，路由层也未传递，导致可改到其他小说的资源。
4. **add_character_to_storyline 未校验归属**：可建立跨小说的故事线-人物关联。

---

## 核心功能点

### F1: ConversationService.get_conversation(conv_id, novel_id) 双键查询
- 修改签名：`get_conversation(self, conv_id: int, novel_id: str) -> dict | None`
- 查询条件：`WHERE id = conv_id AND novel_id = novel_id`
- 所有调用方（send_message、conclude_conversation、generate_from_conversation 等）同步更新

### F2: confirm_message 三元组校验
- 查询消息时同时限定：`msg.conversation_id == conv_id` 且 `conversation.novel_id == novel_id`
- 不满足则抛 ValueError("消息不存在或无权访问")

### F3: 所有 conversation route 传 novel_id
- `send_message`、`conclude_conversation`、`generate_outline_from_conversation` 路由均将 novel_id 传给 service

### F4: arc/scene update/delete service 方法 novel_id 改为必填
- `update_character_arc(arc_id, novel_id, **kwargs)` — novel_id 无默认值
- `delete_character_arc(arc_id, novel_id)` — novel_id 无默认值
- `update_scene(scene_id, novel_id, **kwargs)` — novel_id 无默认值
- `delete_scene(scene_id, novel_id)` — novel_id 无默认值
- storyline 的 update/delete 已有 novel_id，改为必填（去掉 `= None`）

### F5: arc/scene route 传 novel_id
- `update_character_arc`、`delete_character_arc`、`update_scene`、`delete_scene` 路由传 novel_id 给 service

### F6: add_character_to_storyline 校验归属
- 在插入前查询 storyline 是否属于 novel_id（通过 storyline_id 查 Storyline.novel_id）
- 不属于则抛 ValueError

### F7: get_relations 改为 DB 过滤
- StorylineCharacter 查询加 JOIN Storyline WHERE novel_id = novel_id，避免全表扫描后 Python 过滤

### F8: 补充跨小说负例测试
- 跨小说改 arc（用 novel_A 的 arc_id 在 novel_B 路由下 PUT → 404）
- 跨小说改 scene（用 novel_A 的 scene_id 在 novel_B 路由下 PUT → 404）
- 跨小说确认 message（用 novel_A 的 msg_id 在 novel_B 路由下 confirm → 404）

---

## 技术可行性

- 所有修改均在现有 SQLAlchemy + FastAPI 框架内，无新依赖
- 修改 service 签名后需同步更新所有调用方（内部调用约 5 处）
- 测试使用现有 httpx + pytest-asyncio 框架，与 test_routes.py 同结构

---

## 风险

- `send_message` 内部调用 `get_conversation(conv_id)` 两次，需同步传 novel_id（可从 conv_data 中取）
- `generate_from_conversation` 在 storyline_service 中调用 `conv_service.get_conversation(conv_id)`，需传 novel_id

## 验收标准

1. `GET /api/v1/projects/novel_B/conversations/{conv_id_of_novel_A}` → 404
2. `POST /api/v1/projects/novel_B/conversations/{conv_id}/messages/{msg_id_of_novel_A}/confirm` → 404
3. `PUT /api/v1/projects/novel_B/character-arcs/{arc_id_of_novel_A}` → 404
4. `PUT /api/v1/projects/novel_B/scenes/{scene_id_of_novel_A}` → 404
5. ruff 0 errors，mypy 0 errors（针对修改文件）
6. 所有现有测试继续通过

# CHANGE-030 故事知识图谱 — 需求分析

**变更编号**: CHANGE-030  
**日期**: 2026-05-19  
**状态**: 待审批

---

## 1. 需求背景

当前 xIaoShuo 平台在长篇小说生成中面临以下问题：

- **上下文遗忘**：LLM 上下文窗口有限，跨章节的人物关系、伏笔、世界观规则容易丢失
- **一致性缺陷**：角色性格突变、时间线矛盾、已死角色复活等问题频发
- **手动维护成本高**：现有 storylines/characters/world_setting 是静态设定数据，需要人工维护，无法自动跟踪章节内容的演变

现有系统已有：
- `Storyline` / `CharacterArc` / `Scene` — 手动或 AI 一次性生成的静态故事结构
- `RelationGraph.vue` — 基于上述静态数据的 D3 树形可视化
- `Character` / `WorldSetting` — 创作前设定的人物和世界观

**缺失能力**：从已生成章节文本中**自动、增量**地抽取结构化知识（实体+关系三元组），并在后续生成时**动态检索**相关上下文注入 prompt。

---

## 2. 业务目标

| 目标 | 量化指标 |
|------|----------|
| 提升长篇连贯性 | 一致性检查通过率 >= 90%（人物关系、时间线、世界观规则） |
| 降低人工修正成本 | 自动抽取覆盖率 >= 80%（关键实体和关系被正确识别） |
| 不影响生成速度 | 单章抽取耗时 < 10s；上下文检索耗时 < 500ms |
| 增量处理 | 已抽取章节不重复处理，支持章节修改后重新抽取 |

---

## 3. 核心功能点

### 3.1 自动抽取模块（Extraction）

**输入**：单章文本 + 小说元信息（novel_id, chapter_number）  
**输出**：实体列表 + 关系三元组列表

抽取的实体类型：
- **人物**（Character）：姓名、别名、身份、状态（存活/死亡/失踪）
- **地点**（Location）：名称、层级（大陆>国家>城市>具体地点）
- **组织**（Organization）：名称、类型（门派/国家/势力）
- **道具**（Item）：名称、类型（武器/丹药/法宝）、持有者
- **事件**（Event）：名称、时间点、参与者
- **伏笔**（Foreshadowing）：描述、状态（埋设/回收/悬挂）

抽取的关系类型（三元组 subject-predicate-object）：
- 人物-人物：师徒、父子、敌对、恋人、盟友...
- 人物-组织：隶属、创建、背叛...
- 人物-地点：位于、前往、离开...
- 人物-道具：持有、失去、赠予...
- 事件-人物：参与、发起、受害...
- 伏笔-事件：触发、关联...

**触发时机**：
1. 章节生成完成后自动触发（LangGraph 后处理节点）
2. 章节手动编辑保存后触发（API 钩子）
3. 手动触发重新抽取（API 接口）

### 3.2 上下文检索模块（Retrieval）

**输入**：当前章节的写作目标（大纲、涉及人物、场景）  
**输出**：相关三元组 + 实体详情，格式化为 prompt 片段

检索策略：
1. **实体匹配**：从章节大纲中提取涉及的人物/地点名称，查询相关三元组
2. **关系链路**：给定核心人物，查询 1-2 跳关系网络
3. **时间线上下文**：查询最近 N 章的关键事件
4. **伏笔追踪**：查询状态为"埋设"的伏笔，提醒 LLM 适时回收

**注入位置**：章节生成 prompt 中，作为"已知设定"上下文段落

### 3.3 一致性检查模块（Consistency Check）

**输入**：新生成章节文本 + 该章抽取结果 + 历史图谱  
**输出**：冲突列表（severity: error/warning）

检查维度：
1. **人物关系一致性**：新章节中的关系是否与历史矛盾（如已确认的敌对关系突然变为盟友，无合理转折）
2. **时间线一致性**：事件顺序是否合理，已死角色是否复活
3. **世界观规则一致性**：是否违反已建立的世界规则（如修仙等级体系跳级）
4. **伏笔一致性**：已回收的伏笔是否被重复埋设

**触发时机**：章节生成后、抽取完成后自动执行  
**结果处理**：
- error 级别：阻断流程，要求重新生成或人工修正
- warning 级别：记录但不阻断，在 human_review 阶段展示

## 4. 系统边界

### 4.1 与现有系统的关系

| 现有模块 | 知识图谱的关系 | 说明 |
|----------|---------------|------|
| `Character` 表 | 互补 | 图谱中的人物实体引用 Character.id，但图谱额外记录动态状态变化 |
| `WorldSetting` 表 | 互补 | 世界观规则作为一致性检查的基准，图谱不替代它 |
| `Storyline` 表 | 互补 | 故事线是宏观规划，图谱是微观事实记录 |
| `RelationGraph.vue` | 扩展 | 前端可增加"知识图谱"视图 tab，展示三元组网络 |
| LangGraph 流程 | 集成 | 在 chapter_generation 后插入抽取节点，在 chapter_generation 前插入检索节点 |

### 4.2 不做的事情

- 不替代现有 Character/WorldSetting/Storyline 的 CRUD 功能
- 不做跨小说的知识共享（图谱严格隔离到 novel_id）
- 不做实时流式抽取（按章节粒度批量处理）
- 不做自然语言问答接口（仅结构化检索）

## 5. 技术可行性分析

### 5.1 LLM 抽取能力

- DeepSeek 模型支持结构化输出（JSON mode），可通过精心设计的 prompt 实现实体和关系抽取
- 单章文本通常 3000-8000 字，在模型上下文窗口内
- 抽取结果需要 JSON Schema 约束 + 后处理校验

**风险**：LLM 抽取可能遗漏或产生幻觉实体  
**缓解**：与已有 Character 表做交叉验证；支持人工修正接口

### 5.2 存储方案

- PostgreSQL 原生支持 JSON 字段，可存储三元组属性
- 使用关系型表（非图数据库）存储三元组，通过索引实现高效查询
- 预计单部小说三元组量级：100-2000 条（可控）

**不选择图数据库的原因**：
- 项目已有 PostgreSQL，引入 Neo4j 增加运维复杂度
- 三元组规模小（千级），关系型数据库 + 索引足够
- 查询模式固定（按实体/章节/类型），不需要复杂图遍历

### 5.3 检索效率

- 按 novel_id + entity_name 索引，单次查询 < 10ms
- 上下文组装（格式化为 prompt 片段）在应用层完成
- 总检索耗时目标 < 500ms 可达

## 6. 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 抽取准确率不足 | 图谱质量差，检索注入噪声 | 设计校验逻辑 + 人工修正 API |
| 抽取增加生成耗时 | 用户体验下降 | 异步抽取，不阻塞章节返回 |
| 三元组冲突判断误报 | 打断正常生成流程 | warning 不阻断，仅 error 阻断 |
| 与现有 storyline 数据重复 | 用户困惑 | UI 明确区分"规划数据"和"抽取数据" |

**外部依赖**：
- DeepSeek API（已有）
- PostgreSQL（已有）
- 无新增外部依赖

## 7. 验收标准

### 7.1 功能验收

1. 生成一章后，系统自动抽取出实体和关系三元组，存入数据库
2. 生成下一章时，prompt 中包含从图谱检索的相关上下文
3. 一致性检查能检出明显矛盾（如已死角色出现）
4. 提供 API 查询图谱数据，前端可展示三元组网络
5. 支持手动修正（增删改三元组）

### 7.2 性能验收

- 单章抽取耗时 < 10s（含 LLM 调用）
- 上下文检索耗时 < 500ms
- 一致性检查耗时 < 5s（含 LLM 调用）

### 7.3 质量验收

- 抽取覆盖率 >= 80%（人工抽样 10 章验证）
- 一致性检查通过率 >= 90%（无漏报 error 级矛盾）
- 不影响现有生成流程的正常运行（回归测试通过）

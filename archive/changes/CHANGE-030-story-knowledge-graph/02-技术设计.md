```sql
CREATE TABLE knowledge_entities (

    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()

CREATE INDEX idx_ke_novel_id ON knowledge_entities(novel_id);
CREATE INDEX idx_ke_novel_type ON knowledge_entities(novel_id, entity_type);
CREATE INDEX idx_ke_novel_name ON knowledge_entities(novel_id, name);
```

#### `knowledge_triples` — 知识三元组表

```sql
CREATE TABLE knowledge_triples (
```

## 3. 上下文检索机制

### 3.1 检索流程

```
章节生成前（chapter_generation 节点内部）
    ↓
从 chapter_outline 提取关键词（人物名、地点名）
    ↓
查询 knowledge_entities WHERE novel_id = ? AND name IN (关键词)
    ↓
查询 knowledge_triples WHERE subject_id IN (实体IDs) OR object_id IN (实体IDs)
    ↓
查询最近 3 章的 event 类型实体（时间线上下文）
    ↓
查询 foreshadowing_status = 'planted' 的伏笔实体
    ↓
格式化为 prompt 片段
```

### 3.2 检索 SQL 模式

```sql
-- 1. 按人物名查相关三元组
SELECT t.*, 
       s.name AS subject_name, s.entity_type AS subject_type,
       o.name AS object_name, o.entity_type AS object_type
FROM knowledge_triples t
JOIN knowledge_entities s ON t.subject_id = s.id
JOIN knowledge_entities o ON t.object_id = o.id
WHERE t.novel_id = :novel_id
  AND t.status = 'active'
  AND (s.name IN (:names) OR o.name IN (:names))
ORDER BY t.chapter_number DESC
LIMIT 50;

-- 2. 查询悬挂伏笔
SELECT * FROM knowledge_entities
WHERE novel_id = :novel_id
  AND entity_type = 'foreshadowing'
  AND attributes->>'foreshadowing_status' = 'planted';

-- 3. 最近事件
SELECT * FROM knowledge_entities
WHERE novel_id = :novel_id
  AND entity_type = 'event'
  AND last_chapter >= :current_chapter - 3
ORDER BY last_chapter DESC
LIMIT 10;
```

### 3.3 上下文格式化模板

注入到章节生成 prompt 中的格式：

```
## 已知设定（来自知识图谱，请严格遵守）

### 人物关系
- 张三 → 师徒 → 李四（第3章确立）
- 张三 → 敌对 → 王五（第7章确立）

### 当前人物状态

- 第8章：张三突破金丹期
- 第7章：王五偷袭青云山失败
```

### 3.4 检索结果裁剪

为避免注入过多上下文占用 token：
- 三元组最多注入 30 条
- 按 chapter_number 倒序排列（优先最近的）
- 按 confidence 降序排列
- 总注入文本不超过 1500 字

### 4.2 检查维度与实现

#### 4.2.1 规则检查（无需 LLM）

```python
async def _rule_based_check(self, novel_id: str, new_triples: list) -> list[dict]:
    """基于规则的一致性检查"""
    conflicts = []

    # 1. 死亡角色复活检查
    for triple in new_triples:
        subject = await self._get_entity(triple["subject_id"])
        if subject.entity_type == "character":
            if subject.attributes.get("status") == "dead":
                # 检查新三元组是否暗示该角色仍活跃
                if triple["predicate"] in ("位于", "前往", "战斗", "对话"):
                    conflicts.append({
                        "severity": "error",
                        "type": "dead_character_active",
                        "message": f"角色'{subject.name}'已在第{...}章死亡，但在新章节中仍有活动",
                        "entity": subject.name,
                    })

    # 2. 关系矛盾检查（同一对实体的互斥关系）
    EXCLUSIVE_PREDICATES = [("敌对", "盟友"), ("师徒", "敌对")]
    ...

    return conflicts
```

```python
async def node(state: NovelState) -> NovelState:
```

### 6.3 修改 quality_check 节点

文件：`src/core/langgraph/nodes/quality_check.py`

```python
async def node(state: NovelState) -> NovelState:
    # ... 现有质量检查逻辑 ...

    # [新增] 一致性检查
    kg_service = get_knowledge_graph_service()
    last_chapter = state["chapters"][-1] if state["chapters"] else None
    if last_chapter:
        conflicts = await kg_service.check_consistency(
            novel_id=state["project_id"],
            chapter_number=len(state["chapters"]),
            chapter_text=last_chapter["content"],
        )
        error_count = sum(1 for c in conflicts if c["severity"] == "error")
        consistency_score = 1.0 if error_count == 0 else max(0.3, 1.0 - error_count * 0.2)
    else:
        consistency_score = 1.0

    quality_scores = {
        **state.get("quality_scores", {}),
        "consistency": consistency_score,
    }

    return {**state, "quality_scores": quality_scores}
```

```python
async def should_extract(self, novel_id: str, chapter_number: int) -> bool:
    """判断该章节是否需要（重新）抽取"""
    log = await self._get_extraction_log(novel_id, chapter_number)
    if log is None:
        return True  # 从未抽取
    if log.status == "failed":
        return True  # 上次失败
    # 检查章节内容是否在抽取后被修改
    chapter = await self._get_chapter(novel_id, chapter_number)
    if chapter.updated_at > log.created_at:
        return True  # 内容已更新
    return False
```

### 9.2 重新抽取时的处理

- 删除该章节产生的所有三元组（`WHERE chapter_number = ?`）
- 不删除实体（实体可能被其他章节的三元组引用）
- 重新执行抽取流程
- 更新 extraction_log

## 10. 性能考量

| 操作 | 预期耗时 | 优化手段 |
|------|----------|----------|
| 单章抽取 | 3-8s | LLM 调用为主，无法压缩；异步不阻塞 |
| 上下文检索 | 50-200ms | 索引覆盖查询；结果缓存（Redis, TTL 60s） |
| 一致性检查（规则） | < 50ms | 纯内存计算 |
| 一致性检查（LLM） | 2-5s | 仅在规则检查发现疑似问题时触发 |
| 可视化数据查询 | 100-500ms | 按 novel_id 索引；前端缓存 |

## 11. 迁移与兼容性

### 11.1 对现有数据的处理

- 新表独立于现有表，不修改任何现有表结构
- 已有小说可通过 `POST /extract-all` 接口触发全量抽取
- 知识图谱功能为可选增强，不影响无图谱数据的小说正常生成

### 11.2 Feature Flag

```python
# src/core/config.py
KNOWLEDGE_GRAPH_ENABLED: bool = True  # 可通过环境变量关闭
```

当 `KNOWLEDGE_GRAPH_ENABLED = False` 时：
- 章节生成后不触发抽取
- 章节生成前不注入图谱上下文
- 一致性检查跳过图谱维度
- API 路由仍可访问（返回空数据）

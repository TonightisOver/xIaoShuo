## 2. 模块 A：重试逻辑

### 2.1 问题根因

`@retry` 装饰器在类定义时求值，无法访问实例属性 `self.max_retries`。同时 `retry_if_exception_type` 只覆盖网络层异常，不覆盖 HTTP 状态码异常。

### 2.2 设计方案

**方案选择**: 将 `@retry` 装饰器移除，改为在方法体内使用 `tenacity.AsyncRetrying` 上下文管理器，在 `__init__` 时读取 `max_retries`。

**HTTP 状态码异常处理**: `langchain_openai` 底层使用 `httpx`，HTTP 错误会抛出 `httpx.HTTPStatusError`。需要自定义 `retry_if_exception` 谓词，检查状态码是否为 429 或 5xx。

### 2.3 改动点：`src/core/llm/client.py`

```
改动说明：
1. 移除顶层 @retry 装饰器及其导入（stop_after_attempt, retry_if_exception_type）
2. 新增导入：tenacity.AsyncRetrying, tenacity.retry_if_exception, httpx
3. __init__ 中保存 self.max_retries（已有）
4. generate() 方法改为：
   - 用 AsyncRetrying(stop=stop_after_attempt(self.max_retries), ...) 包裹 ainvoke 调用
   - retry 谓词：lambda e: isinstance(e, (TimeoutError, ConnectionError)) 
                  or (isinstance(e, httpx.HTTPStatusError) and e.response.status_code in (429, 500, 502, 503))
   - wait: wait_exponential(multiplier=1, min=2, max=30)，429 时等待更长
   - reraise=True
5. 日志中记录重试次数（结合模块 E）
```

**关键设计决策**:
- 401 不重试（认证失败重试无意义，会消耗配额）
- 429 重试时 wait 上限设为 30s（DeepSeek 限流通常在 10-20s 内恢复）
- 5xx 重试（服务端临时故障）

## 5. 模块 D：内存泄漏修复

### 5.1 根因分析

`src/api/services/novel_generator.py` 中：
- 第 120 行：`register_progress_callback(task_id, _chapter_progress_callback)` 注册回调
- 正常完成路径（第 157-168 行）：无 `unregister_progress_callback` 调用
- 异常路径（第 175-185 行）：无 `unregister_progress_callback` 调用

两条路径都不清理，导致每次任务执行后 `_progress_callbacks` 都有残留。

### 5.2 改动点：`src/api/services/novel_generator.py`

```
在 register_progress_callback(task_id, ...) 之后，
将后续所有代码包裹在 try/finally 块中：

try:
    # ... 现有的 graph.astream 循环和 complete_task 调用 ...
    pass
finally:
    unregister_progress_callback(task_id)
```

注意：外层已有 `try/except Exception`，`finally` 需要嵌套在内层，确保 `unregister` 在 `fail_task` 之后执行（避免 fail_task 中还需要用到 callback）。

**具体结构**:
```python
register_progress_callback(task_id, _chapter_progress_callback)
try:
    # graph.astream 循环
    # complete_task
    # _persist_to_novel
    # publish COMPLETED event
except Exception as e:
    # fail_task
    # publish ERROR event
finally:
    unregister_progress_callback(task_id)
```

同样的修复需应用到 `generate_novel_full_background` 函数（如果存在相同模式）。

## 6. 模块 E：结构化日志

### 6.1 方案选择

引入 `structlog`（版本 `>=24.0.0`）。`structlog` 可以与标准 `logging` 共存，通过 `structlog.stdlib.ProcessorFormatter` 接管标准 logging 的输出格式。

**开发模式**: 输出带颜色的人类可读格式（`ConsoleRenderer`）  
**生产模式**: 输出 JSON（`JSONRenderer`），通过 `LOG_FORMAT=json` 环境变量切换

### 6.2 改动点

**`src/core/config.py`**:
```
新增字段：
LOG_FORMAT: str = "console"  # "console" | "json"
```

**`src/core/logging_config.py`**:
```
完全重写 setup_logging()：
1. 配置 structlog processors 链：
   - structlog.contextvars.merge_contextvars_context
   - structlog.stdlib.add_log_level
   - structlog.stdlib.add_logger_name
   - structlog.processors.TimeStamper(fmt="iso")
   - structlog.processors.StackInfoRenderer()
   - structlog.processors.format_exc_info
   - 根据 log_format 选择 JSONRenderer 或 ConsoleRenderer
2. 配置 structlog.configure()
3. 保留标准 logging basicConfig（兼容第三方库）
4. 导出 get_logger() 工厂函数
```

**`src/core/llm/client.py`**:
```
将 logger = logging.getLogger(__name__)
改为 logger = structlog.get_logger(__name__)

将 logger.info(f"Calling LLM API with prompt length: {len(prompt)}")
改为 logger.info("llm_call_start", prompt_length=len(prompt))

将 logger.info(f"LLM API response length: {len(response.content)}")
改为 logger.info("llm_call_complete", response_length=len(response.content))

将 logger.error(f"LLM API call failed: {e}", exc_info=True)
改为 logger.error("llm_call_failed", error=str(e), exc_info=True)
```

**`src/api/services/novel_generator.py`**:
```
将 logger = logging.getLogger(__name__)
改为 logger = structlog.get_logger(__name__)

关键日志点改为结构化：
logger.info("novel_generation_start", task_id=task_id)
logger.info("novel_generation_complete", task_id=task_id, chapters=len(...))
logger.exception("novel_generation_failed", task_id=task_id)
```

**`requirements.txt` 或 `pyproject.toml`**:

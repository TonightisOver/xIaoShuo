## 2. 代码质量

### 2.1 重试逻辑（client.py）

**评价**: `AsyncRetrying` 上下文管理器用法正确，`reraise=True` 确保最终异常透传。401 提前 raise 绕过重试逻辑正确。

**Warning W-03**: `_is_retryable_http_error()` 函数定义了但在 `retry=retry_if_exception_type(...)` 中未使用——`retry_if_exception_type` 只按类型过滤，不调用该函数。实际上 `httpx.HTTPStatusError` 的所有实例（包括 404、400）都会触发重试，直到 401 的提前 raise 阻止。应改用 `retry=retry_if_exception(lambda e: _is_retryable_http_error(e) or isinstance(e, (TimeoutError, ConnectionError)))` 以精确控制重试条件。这是逻辑缺陷，但因 401 已提前 raise，实际影响有限。

**Warning W-04**: `wait_exponential(multiplier=1, min=2, max=30)` 在测试中会导致真实等待，建议测试时通过 mock 覆盖 wait 策略（现有测试已用 `side_effect` 绕过，但文档未说明）。

### 2.2 内存泄漏修复（novel_generator.py）

**评价**: `generate_novel_background` 和 `_run_langgraph_pipeline` 均已用 `try/finally` 包裹，`finally` 调用 `unregister_progress_callback`，修复正确。

**Warning W-05**: `generate_novel_background` 中 `register_progress_callback` 在第 121 行，位于 `try` 块内部（第 51 行 `try` 开始），但在 `initial_state` 构建之后。若 `initial_state` 构建阶段（第 84-119 行）抛出异常，`register_progress_callback` 不会被调用，`finally` 中的 `unregister_progress_callback` 也是空操作——这是安全的，无泄漏风险。逻辑正确。

### 2.3 数据库索引（db_models.py + migration）

**评价**: `Chapter.status`、`Volume.status`、`Conversation.status`、`Outline.status`、`Storyline.status` 均加 `index=True`，migration 文件与模型定义一致，`down_revision` 正确指向前一个 revision。

**Warning W-06**: `Novel.status` 在 `db_models.py` 已有 `index=True`（第 27 行），但 migration 文件未包含 `novels.status` 的索引创建。这说明 `novels.status` 索引可能已在更早的 migration 中创建，或依赖 `create_all` 自动创建。需确认历史 migration 中是否已覆盖，避免重复创建或遗漏。

### 2.4 novel_manager.py

**评价**: `list_novels` 注释说明无 N+1 问题，分析正确——`_novel_summary()` 只访问标量列，不触发关系懒加载。

`cache_logger_on_first_use=True` 在多进程/多线程场景下可能导致 logger 在 fork 前被缓存，但本项目为单进程 asyncio，无风险。

`get_settings()` 使用 `lru_cache`，测试中需调用 `get_settings.cache_clear()` 重置——`conftest.py` 已正确处理。

## 2. 问题清单与分析

### 问题 1：重试逻辑硬编码且异常覆盖不足

**文件**: `src/core/llm/client.py`

**现状**:
- `@retry` 装饰器使用 `stop_after_attempt(3)` 字面量，与 `self.max_retries = settings.DEEPSEEK_MAX_RETRIES` 完全脱节——修改配置不生效。
- `retry_if_exception_type((TimeoutError, ConnectionError))` 只捕获网络层异常，不捕获 HTTP 429（限流）和 401（认证失败）。DeepSeek API 在高并发时会返回 429，当前实现会直接抛出而不重试。

**影响**: 配置项 `DEEPSEEK_MAX_RETRIES` 形同虚设；429 错误不重试导致任务失败率偏高。

**约束**: `@retry` 装饰器是静态的，不能直接读取实例属性。需要改为在 `__init__` 中动态构建重试逻辑，或使用 `tenacity.Retrying` 上下文管理器。

### 问题 2：CORS 配置违反规范

**文件**: `src/api/main.py`

**现状**:
```python
allow_origins=["*"],
allow_credentials=True,
```

**影响**: 浏览器会拒绝此配置。CORS 规范明确禁止同时使用通配符 origin 和 `credentials: true`——浏览器会抛出 CORS 错误，导致前端无法携带 Cookie 或 Authorization header 发起跨域请求。

**约束**: 修复方案有两种：
1. 去掉 `allow_credentials=True`（适合纯 token 认证，无 Cookie）
2. 将 `allow_origins` 改为明确的域名列表（适合生产环境）

当前项目为开发阶段，前端通过 Vite dev server 访问，推荐方案 1（去掉 credentials），同时保留通配符 origin，并在配置中预留生产环境 origins 的扩展点。

### 问题 4：进度回调内存泄漏

**文件**: `src/api/services/progress_event_bus.py`

**现状**:
```python
_progress_callbacks: dict[str, Any] = {}
```
`register_progress_callback` 在任务开始时注册，但 `novel_generator.py` 的异常路径（`except Exception`）只调用了 `task_manager.fail_task` 和 `event_bus.publish`，**没有调用 `unregister_progress_callback`**。

经核查 `novel_generator.py`（第 175-185 行），`finally` 块不存在，`unregister_progress_callback` 只在正常完成路径中被调用（实际上也没有——代码中 `register_progress_callback` 在第 120 行注册，但整个函数没有对应的 `unregister` 调用）。

**影响**: 每个失败或异常终止的任务都会在 `_progress_callbacks` 中留下一个永不清理的条目。长期运行后内存持续增长。

**约束**: 修复需在 `novel_generator.py` 的 `finally` 块中调用 `unregister_progress_callback`，确保无论成功/失败都清理。

### 问题 5：路由缺少 response_model

**文件**: `src/api/routes/` 下所有路由文件

**现状**（经代码核查）:
- `novels.py`：3 个路由有 `response_model`（`TaskResponse`、`TaskDetailResponse`、`TaskListResponse`）✅
- `health.py`：1 个路由有 `response_model`（`HealthResponse`）✅
- `projects.py`：约 30 个路由，**全部无 `response_model`**
- `outlines.py`：约 10 个路由，**全部无 `response_model`**
- `conversations.py`：约 10 个路由，**全部无 `response_model`**
- `storylines.py`：约 20 个路由，**全部无 `response_model`**
- `style.py`：1 个路由，**无 `response_model`**

**影响**: OpenAPI 文档中响应 schema 为空，前端无法自动生成类型；FastAPI 不做响应序列化校验，可能泄露内部字段。

**约束**: 补全 response_model 需要先定义对应的 Pydantic 响应模型。考虑到路由数量（约 68 个），本次优先覆盖高频核心路由（projects CRUD、chapters、volumes），其余路由补充 `dict` 类型注解作为过渡。

### 问题 6：N+1 查询

**文件**: `src/api/services/novel_manager.py`

**现状**（经代码核查）:
```python
async def list_novels(self, ...):
    query = select(Novel)
    ...
    novels = (await session.execute(query)).scalars().all()
    return [self._novel_summary(n) for n in novels], total
```

`_novel_summary` 只访问 `Novel` 自身字段（`novel_id`, `title`, `novel_type`, `status`, `target_words`, `created_at`, `updated_at`），**不访问关联关系**，因此 `list_novels` 本身不存在 N+1 问题。

但 `get_novel` 调用 `_novel_to_dict`，后者访问 `novel.world_setting`、`novel.characters`、`novel.power_systems`，这些已通过 `selectinload` 预加载 ✅。

真正的 N+1 风险在于：`projects.py` 中多个路由先调用 `get_novel`（触发一次完整加载含 selectinload），再调用 `list_chapters`/`list_volumes` 等（各自独立查询）。这是合理的分层设计，不是 N+1。

**结论**: 原始问题描述中的 N+1 在当前代码中已部分处理。真正需要优化的是 `list_novels` 在返回 summary 时如果将来需要包含 `chapters_count` 等聚合字段，需要用 `func.count` 子查询而非 lazy load。当前 `_novel_summary` 不含此类字段，暂无 N+1 问题。

**本次处理**: 确认现状，在 `list_novels` 中添加注释说明，不做无谓改动。

### 问题 7：结构化日志缺失

**文件**: `src/core/logging_config.py` 及全部使用 `logger.info(f"...")` 的文件

**现状**: 使用标准 `logging` 模块 + f-string 格式，输出为纯文本。无法按字段（task_id、novel_id、stage 等）过滤或聚合。

**影响**: 生产环境排查问题时，只能 grep 文本，无法用日志平台（ELK、Loki 等）做结构化查询。

**方案**: 引入 `structlog` 库，替换 `logging_config.py` 的配置，并在关键调用点（`novel_generator.py`、`client.py`）改用结构化调用方式 `logger.info("event", key=value)`。

**约束**: `structlog` 与标准 `logging` 兼容，可以渐进式迁移。本次只改 `logging_config.py` 和最关键的两个文件，其余文件保持现状（f-string 日志在 structlog 下仍可工作）。

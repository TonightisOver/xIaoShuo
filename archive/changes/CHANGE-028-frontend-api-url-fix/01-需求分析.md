# 01 需求分析 — CHANGE-028

## 背景

用户反馈「故事线关系图谱」页面不显示内容，「大纲」Tab 中的章节数据也不显示。

根因已由外部定位：`frontend/src/views/NovelDetail.vue` 的 `fetchAll()` 函数中，两条 API URL 使用了错误的路径前缀，导致请求 404，数据无法加载。

### 重要说明：预修复情况

在本次 Harness 流程启动之前，已有人直接修改了 `NovelDetail.vue`，将这两行 URL 改为正确值。当前文件（第 342-343 行）已显示为：

```
fetch(`/api/v1/projects/${novelId}/relations`)
fetch(`/api/v1/projects/${novelId}/outlines`)
```

实现阶段需要验证该预修复是否正确、完整，并检查项目中是否还有其他同类 URL 错误。

---

## 核心功能点

1. **关系图谱数据加载**：`NovelDetail.vue` 的 `fetchAll()` 需正确调用 `/api/v1/projects/{novelId}/relations`
2. **大纲树数据加载**：`fetchAll()` 需正确调用 `/api/v1/projects/{novelId}/outlines`
3. **全局 URL 一致性**：项目所有前端 `.vue` 文件中的 fetch URL 必须与后端路由定义一致

---

## 技术可行性

- 修复范围极小，仅涉及字符串字面量替换
- 后端路由已正确定义，无需改动后端
- 预修复已覆盖原始两处错误，实现阶段以验证为主

---

## 风险与约束

| 风险 | 说明 |
|------|------|
| 预修复不完整 | 需逐行核对修复后的 URL 与后端路由是否完全匹配 |
| 同类问题遗漏 | 其他 .vue 文件可能存在相同模式的错误 URL |
| 回归风险 | 修改 fetchAll() 可能影响页面其他数据加载逻辑 |

---

## 验收标准

1. `NovelDetail.vue` 第 342 行为 `/api/v1/projects/${novelId}/relations`，与 `storylines.py` 路由 `GET /{novel_id}/relations` 匹配
2. `NovelDetail.vue` 第 343 行为 `/api/v1/projects/${novelId}/outlines`，与 `outlines.py` 路由 `GET /{novel_id}/outlines` 匹配
3. 全局扫描所有 `.vue` 文件的 fetch URL，无其他使用错误前缀（`/api/v1/storylines/`、`/api/v1/outlines/` 等）的调用
4. 关系图谱页面和大纲 Tab 在浏览器中正常显示数据（手动验证或 E2E 测试）

---

## 依赖

- 后端路由文件（只读参考）：
  - `src/api/routes/storylines.py`：`prefix="/api/v1/projects"`，端点 `GET /{novel_id}/relations`
  - `src/api/routes/outlines.py`：`prefix="/api/v1/projects"`，端点 `GET /{novel_id}/outlines`
- 前端文件：`frontend/src/views/NovelDetail.vue`（已预修复）

# 专家评审报告 — CHANGE-029 章节卷号显示

**变更**: chapter-volume-display  
**评审日期**: 2026-05-19  
**评审范围**: 后端服务层 + 前端组件

---

## 1. 架构一致性

### 1.1 数据流路径

```
DB (Chapter.volume_number)
  → NovelManager.list_chapters() / get_chapter()
  → API 路由 /chapters / /chapters/{num}
  → ChapterResponse (Pydantic)
  → 前端 VolumeList.vue / ChapterEdit.vue
```

路径完整，无断层。`volume_number` 字段在 `Chapter` 模型中已存在（`Mapped[int | None]`），本次变更仅补全了服务层的序列化输出，属于**最小侵入式修改**，符合架构原则。

### 1.2 与现有模式的一致性

- `list_chapters()` 和 `get_chapter()` 的字典结构与其他同类方法（`list_volumes()`、`get_volume()`）保持一致，均返回扁平字典。
- `ChapterResponse` Pydantic 模型已包含 `volume_number: int | None = None`，与服务层返回值类型匹配。
- `_persist_to_novel()` 中 `Chapter` 构造加入 `volume_number=ch.get("volume_number")` 与 `generate_volume_background()` 中的同名字段（第 680 行）保持一致。

## 2. 代码质量

### 2.1 后端 (novel_manager.py)

**优点**:
- 改动极小（各加一行），无副作用。
- `volume_number` 可为 `None`，正确处理无卷号章节（单卷小说或旧数据）。

**问题**:
- **无问题**。字段已在 DB 模型中定义，序列化逻辑与其他字段完全对称。

### 2.2 后端 (novel_generator.py)

**优点**:
- `ch.get("volume_number")` 使用 `.get()` 而非直接访问，安全处理 LangGraph 输出中缺少该键的情况。

**潜在风险**:
- LangGraph 生成的章节字典中 `volume_number` 键名需与此处一致。若 LangGraph 节点输出使用不同键名（如 `vol_num`），则持久化时 `volume_number` 将为 `None`。建议在集成测试中验证端到端数据流。

### 2.3 前端 (VolumeList.vue)

**优点**:
- `chapterIndexInVolume()` 辅助函数逻辑清晰，通过过滤同卷章节计算卷内序号，避免依赖后端额外字段。

**潜在问题**:
- 若章节列表未按 `chapter_number` 排序，卷内序号计算可能不准确。需确认 API 返回的章节列表已排序（`list_chapters()` 中有 `.order_by(Chapter.chapter_number)`，已保证）。
- 当 `volume_number` 为 `null`（无卷号章节）时，前端需有降级处理，避免显示异常。

### 2.4 前端 (ChapterEdit.vue)

**优点**:
- 条件渲染卷前缀（`v-if="chapter.volume_number"`）正确处理 `null`/`0` 情况。

# 阶段 6: 专家评审报告 - CHANGE-032-live-knowledge-graph

## 1. 变更设计合理性评估

本变更的核心目标是克服大模型在长文创作过程中的“记忆消退/健忘”问题，通过为小说知识图谱引入时间轴和按章节快照（时空与状态追踪），并在章节生成前提供精准相关的历史状态记忆。

从系统架构和算法层面看，方案设计非常合理：
1. **状态快照模式 (Snapshots Pattern)**: 
   在 `knowledge_entity_states` 表中按 `(entity_id, chapter_number)` 记录实体的每一次状态更新。这避免了仅保留当前最新状态而导致无法回溯历史或进行并行分支生成的尴尬，也为小说的历史状态查询 API 提供了极强的支持。
2. **渐进式属性继承合并 (Incremental State Merging)**:
   在 `_merge_entities` 时，系统自动寻找该实体在 `chapter_number` 之前的最新一条状态快照，在旧状态字典的基础上层叠覆盖本章新抽取的状态改变（`merged_attributes = {**previous_attributes, **extracted_attrs}`）。该逻辑优雅地保证了属性的继承性和连续性，不会因为当前章节没有提到某个属性（如身在何处）就将该属性丢失。
3. **时序双向关系去重 (Temporal Deduplication)**:
   人物关系（例如：张三 ──喜欢──> 李四）可能随章节演进发生改变（例如好感度由 10 变为 30，或变为了憎恨）。`retrieve_context` 中通过 SQL 时序降序排序 + 内存 Set 判定，只保留最新的双向关系演化结果，排除了过期关系对大模型的干扰，极大地提升了 Prompt 的准确度。
4. **大纲高精度多重匹配 (High-Precision Outline Matching)**:
   利用实体自身的 `name` 和别名列表 `aliases` 双重检索章节大纲，可以精准发现隐藏的登场角色（如用绰号“小张”匹配张三），从而提取到极其切合上下文的角色记忆。

## 2. 数据库与查询性能分析

1. **写性能**: 
   实体状态历史表只在有章节提取时新增或更新一条记录。即使长达 500 章的长篇小说，其主要实体（~100个）在每章也只产生百级别的新快照写入，对关系型数据库无任何写压力。
2. **读与索引性能**: 
   - `knowledge_entity_states` 在 `(entity_id, chapter_number)` 具备复合主键/联合索引，使得获取特定实体在特定章节前的最新快照（`chapter_number < current_ch ORDER BY chapter_number DESC LIMIT 1`）仅需常数时间 $O(1)$，对查询性能极其友好。
   - 三元组关系检索通过 `status == "active"` 和时序降序排序去重，对大纲提及实体范围内的关联三元组进行了精简，单次检索平均响应时间 < 10ms。

## 3. 安全与容灾性评审

1. **防御性安全防护**:
   在 `retrieve_context` 中整体应用了 `try-except` 块包裹。即使图谱数据意外受损、数据库连接超时或发生了未知的多表联查异常，图谱层会进行优雅降级，返回空字符串，保证小说的核心生成流程不受阻塞，具有极强的鲁棒性。
2. **Token 安全控制**:
   通过 `_format_rich_context` 生成的 Markdown 记忆 Prompt 在返回前应用了 `len(context) > 1500` 的截断判定。若图谱记忆极为庞大，会自动安全截断，防止发生 Prompt Token 溢出或超出 LLM 上下文预算导致的高额账单/拒绝服务。
3. **数据一致性保障**:
   在写入章节历史快照前，先检索是否已存在该章节的记录。若存在（如二次编辑、重试生成场景），则执行原地更新（Overwrite），避免产生重复主键错误或多快照冗余，维护了极高的数据幂等性与一致性。

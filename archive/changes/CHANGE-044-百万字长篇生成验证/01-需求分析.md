# 需求分析 -- CHANGE-044-百万字长篇生成验证

**日期**: 2026-05-25
**版本**: v1

## 需求概述

当前系统已完成基础功能开发与 API 压力测试，具备中篇（约20万字）小说的完整生成能力。本需求要求将生成规模从中篇升级为百万字级别长篇网文，验证系统在大规模连续生成场景下的各子系统（StoryBible、知识图谱、人物弧光、伏笔管理、质量评分、章节版本系统）是否能长期维持一致性，并具备识别和处理常见长篇问题（无效注水、人物漂移、设定冲突、主线停滞）的能力。

核心目标不是生成质量本身，而是 **系统在百万字规模下的工程健壮性和一致性维护能力**。

## 核心功能点

### F1. 章节字数精确控制

- 支持单章 2000~4000 字、默认 3000 字左右的章节长度控制
- 当前系统 `max_tokens=8000` 无字数约束，LLM 输出长度波动大
- 需要通过 prompt 指令 + 后置截断/续写确保字数稳定

### F2. 长线结构规划（卷-章二级大纲）

- 支持 8~12 卷、250~400 章的结构
- 当前 `outline_generation` 节点通过单次 LLM 调用生成大纲，百万字规模下单次调用难以产出完整大纲
- 需要分卷规划、逐步细化的两级大纲生成策略

### F3. 章节功能分类

- 区分主线推进章、高潮章、余波章、日常章、铺垫章等章节功能类型
- 每章大纲需包含 `chapter_type` 字段，不同类型对应不同的生成策略和质量评估权重
- 高潮章需要更高冲突度，铺垫章允许低推进度

### F4. 流式分批生成与断点恢复

- 百万字约 300+ 章，不可能一次性生成（内存、LLM 超时、网络波动）
- 需要按卷/批次生成，支持暂停-恢复
- 当前 LangGraph pipeline 一次性跑完全流程，不适合百万字规模

### F5. 长期一致性监控仪表盘

- 连续生成 100 万字过程中，实时追踪以下指标：
  - 质量评分趋势（8维度分值按章节的变化曲线）
  - 知识图谱实体漂移检测（角色状态突变）
  - 伏笔生命周期（种下-悬挂-回收时间线）
  - 主线推进效率（每卷主线推进章占比）
  - 人物出场频率（防止长期失踪）
- 输出每卷级别的质量报告

### F6. 无效注水识别与干预

- 自动检测连续低 `advancement` 评分章节（如连续3章 advancement < 0.5）
- 检测重复描写（readability 评分持续走低）
- 主线停滞检测（连续 N 章无主线事件推进）
- 输出注水报告，支持人工决策是否重写

## 技术可行性分析

### 当前架构支持的部分

1. **StoryBible 约束系统** -- 精准抽取、反向更新、冲突检测已实现，可直接复用
2. **知识图谱** -- 实体/三元组抽取、一致性检查已实现，但需要针对大规模优化（实体数量会指数增长）
3. **质量评分** -- 8维度评分已实现，但缺少跨章节趋势分析
4. **章节版本系统** -- 多版本管理已实现，支持重写和回滚
5. **按卷生成** -- `generate_volume_background` 已实现按卷生成流程

### 需要新增/改造的部分

1. **大纲生成重构** -- 当前单次 LLM 调用无法产出 300+ 章完整大纲，需要分卷细化
2. **章节字数控制** -- 需要在 prompt 中加入字数约束，增加后置字数校验
3. **章节类型系统** -- 大纲模型需要新增 `chapter_type` 字段
4. **分批生成编排** -- 需要新增按卷/批次生成的编排逻辑，支持暂停恢复
5. **质量趋势分析** -- 需要新增跨章节的质量趋势追踪和异常检测
6. **注水检测** -- 需要新增基于连续评分的注水识别服务

### 不需要改动的部分

1. LLM 客户端调用层（`llm/client.py`）
2. 数据库基础模型（`db_models.py`）
3. API 路由框架（`routes/`）
4. 事件总线（`progress_event_bus.py`）

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 单次生成大纲质量不足 | 300+ 章大纲需要分卷细化，跨卷一致性难以保证 | 分级大纲策略：总纲 -> 卷纲 -> 章纲，每级注入上级约束 |
| 百万字生成时间过长 | DeepSeek API 限流/超时，300+ 章可能需要数小时 | 按卷断点生成，单卷失败不影响全量，支持重试 |
| 知识图谱实体爆炸 | 百万字产生数千实体/三元组，查询性能下降 | 实体去重合并、按卷裁剪历史上下文、设置 KG 上下文长度上限 |
| 内存压力 | NovelState 中 chapters 列表存储所有章节内容 | 按卷批量生成，每卷完成后持久化清空内存，仅保留摘要 |
| 字数控制与文笔平衡 | 强制字数限制可能影响生成质量 | prompt 引导为主，后置截断为辅，允许 20% 浮动 |
| 伏笔遗忘累积 | 10章阈值在300+章中过于宽松 | 根据卷长度动态调整阈值，卷末强制检查所有悬挂伏笔 |

## 验收标准

### 量化指标

- [ ] 单章字数控制：95% 的章节字数落在 2000~4000 字区间
- [ ] 大纲完整性：生成的总纲 + 卷纲 + 章纲覆盖 250~400 章
- [ ] 章节类型标注：100% 章节大纲包含 `chapter_type` 字段
- [ ] 分批生成：支持按卷生成，单卷生成失败不影响其他卷
- [ ] 断点恢复：中断后可从指定卷/章恢复生成
- [ ] 质量趋势报告：每卷生成后输出包含 8 维度均值/趋势的质量报告
- [ ] 注水检测：能识别连续3章低 advancement（<0.5）并输出警告
- [ ] 伏笔追踪：能输出所有悬挂伏笔的种下章节、已悬挂章数、关联人物
- [ ] 人物出场统计：能输出所有角色的出场章节列表和出场频率

### 功能验收

- [ ] 通过 API 创建百万字小说任务，指定 8 卷 300 章的结构
- [ ] 系统自动生成总纲 -> 卷纲 -> 章纲的三级大纲
- [ ] 按卷逐卷生成章节内容，每卷生成完成后自动持久化
- [ ] 每章生成后自动触发 StoryBible 反向更新和 KG 知识抽取
- [ ] 每章生成后自动触发质量评分并持久化到 ChapterVersion
- [ ] 全部章节生成完成后，输出完整质量报告
- [ ] 生成过程中可暂停任意卷，恢复后从断点继续

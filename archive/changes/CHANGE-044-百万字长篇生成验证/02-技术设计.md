# 技术设计 -- CHANGE-044-百万字长篇生成验证

**日期**: 2026-05-25
**版本**: v1

## 架构方案

### 整体架构变更

当前系统采用单一 LangGraph pipeline 一次性完成全部生成，不适用于百万字规模。改造为 **三级大纲 + 按卷流式生成 + 跨卷质量监控** 架构：

```
用户请求（target_words=1,000,000, volumes=8, chapters_per_vol=35~50）
    |
    v
[Stage 0] 总纲生成（master_outline_generation）
    |  输出：总纲 JSON（8卷概述 + 主线规划 + 伏笔分布）
    v
[Stage 1] 卷纲细化（volume_outline_generation） x N 卷
    |  输入：总纲 + 前序卷摘要
    |  输出：每卷 35~50 章的章纲（含 chapter_type）
    v
[Stage 2-7] 卷内 LangGraph 7 节点流水线（逐卷执行）
    |  每卷独立执行 idea -> world -> char -> outline -> chapter -> quality -> review
    |  每章完成后：StoryBible 反向更新 + KG 抽取 + 质量评分持久化
    v
[Stage 8-13] 卷后子功能（power_systems, storylines, arcs, scenes, etc.）
    |
    v
[Monitor] 跨卷质量监控（每卷完成后触发）
    |  输出：质量趋势报告 + 注水检测 + 伏笔追踪
    v
全部完成 -> 最终报告
```

### 数据流（单卷内部）

```
卷纲（35~50章大纲）
    |
    v
逐章生成循环 {
    1. StoryBible 精准约束抽取（extract_relevant_constraints）
    2. KG 上下文检索（retrieve_context）
    3. 章节规划 prompt（CHAPTER_PLANNING_PROMPT）
    4. 章节生成 prompt（CHAPTER_GENERATION_PROMPT，含字数约束）
    5. 字数校验与调整
    6. StoryBible 反向更新（update_bible_after_generation）
    7. KG 知识抽取（extract_from_chapter）
    8. 质量评分（quality_check node，8维度）
    9. 评分持久化（_persist_quality_to_version）
}
    |
    v
卷完成后：质量报告生成
```

## 关键决策

### D1. 大纲分级策略

**决策**: 采用三级大纲策略 -- 总纲 -> 卷纲 -> 章纲

**理由**:
- 当前 `outline_generation` 单次 LLM 调用无法产出 300+ 章完整大纲
- DeepSeek API 单次输出 token 上限约 4000，不够输出完整百万字大纲
- 分级策略允许每级注入上级约束，保证跨卷一致性

**实现**:
1. `master_outline_generation` -- 生成总纲（卷级概述 + 主线规划 + 伏笔分布）
2. `volume_outline_generation` -- 逐卷细化（注入总纲 + 前序卷摘要）

### D2. 按卷流式生成而非全量一次性生成

**决策**: 按卷独立执行 LangGraph 流水线，每卷完成后持久化并释放内存

**理由**:
- 百万字内容不可能全部存入 NovelState 内存
- 按卷生成支持暂停/恢复/重试
- 单卷失败不影响其他卷

**实现**:
- 复用现有 `generate_volume_background` 的按卷生成模式
- 每卷生成完成后更新 `LongFormProgress` 状态
- 每章完成后立即持久化到 Chapter + ChapterVersion 表

### D3. 章节类型系统

**决策**: 在章纲中引入 `chapter_type` 字段，区分 6 种章节功能类型

**类型定义**:
| 类型 | 代码 | 说明 | 质量权重调整 |
|------|------|------|-------------|
| 主线推进章 | main_advance | 推进核心剧情 | advancement 高权重 |
| 高潮章 | climax | 冲突爆发、高潮段落 | conflict 高权重 |
| 余波章 | aftermath | 高潮后的调整、影响 | pacing 允许放慢 |
| 日常章 | daily | 日常互动、人物塑造 | character_consistency 高权重 |
| 铺垫章 | setup | 伏笔种下、背景铺设 | foreshadowing 高权重 |
| 注水章 | filler | 系统检测标记 | 标记为低质量，可能触发重写 |

### D4. 字数控制策略

**决策**: prompt 引导为主 + 后置校验为辅，允许 20% 浮动

**理由**:
- LLM 无法精确控制字数，强制截断会破坏文笔
- prompt 中明确指定 "本章约3000字" 可有效约束输出长度
- 后置校验：低于 2000 字则续写，超过 4000 字则标记警告

**实现**:
- `CHAPTER_GENERATION_PROMPT` 新增字数约束指令
- `generate_single_chapter` 返回后校验 `word_count`
- 低于下限时：追加续写 prompt
- 超过上限时：记录警告但不截断（保留完整性）

### D5. 质量趋势监控

**决策**: 每卷完成后生成卷级质量报告，全书完成后生成总报告

**监控维度**:
- 8 维度评分的卷内均值和标准差
- 按章节的评分变化趋势
- 连续低分检测（连续 3 章 advancement < 0.5）
- 伏笔生命周期统计
- 人物出场频率统计
- 主线推进效率（main_advance 类章节占比）

### D6. 知识图谱上下文裁剪

**决策**: 百万字场景下对 KG 上下文进行智能裁剪，限制 prompt 长度

**理由**:
- 百万字产生数千实体/三元组，全量注入会超出 LLM 上下文窗口
- 需要按相关性筛选

**实现**:
- `retrieve_context` 已有 1500 字符上限
- 新增按卷裁剪：每卷只注入当前卷相关实体 + 前序卷摘要
- 伏笔列表保留全部悬挂伏笔（不超过 20 条）
- 人物关系只注入当前章出场人物的直接关系

### 性能影响

- **数据库**: 新增 LongFormProgress 表，百万字生成约产生 300+ Chapter 记录、300+ ChapterVersion 记录、10+ Volume 记录、1 条 LongFormProgress 记录。数据量级在 PostgreSQL 承受范围内。
- **LLM API**: 百万字生成需要 300+ 次章节生成调用 + 300+ 次质量检查调用 + 300+ 次 KG 抽取调用 + 300+ 次 StoryBible 更新调用，总计约 1500+ 次 LLM 调用。按 DeepSeek 30 QPS 限制，预计耗时 2~4 小时。
- **内存**: 按卷生成模式下，内存峰值为单卷（40章 x 3000字）的章节内容 + 上下文，约 50MB，完全可控。
- **数据库连接**: 按卷生成需要频繁的数据库读写（每章 5+ 次），建议增大 `DATABASE_POOL_SIZE` 到 20。

### Alembic 迁移

需要新增一个 Alembic 迁移脚本：
- `chapters` 表新增 `chapter_type` 字段
- `volumes` 表新增 `target_chapters`、`generated_chapters`、`avg_quality_score`、`quality_report` 字段
- `novels` 表新增 `is_long_form`、`total_volumes`、`chapters_per_volume`、`words_per_chapter`、`master_outline` 字段

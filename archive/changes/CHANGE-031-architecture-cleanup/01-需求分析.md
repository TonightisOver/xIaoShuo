# CHANGE-031 需求分析

## 变更概述

基于全项目架构审查，修复 3 个 Critical 和 5 个 High 级别问题。核心目标：消除重复代码、统一日志框架、补全数据完整性约束。

---

## 问题验证与现状确认

### C1. 初始状态构建逻辑重复 3 次

**已验证位置：**
- `novel_generator.py` 第 84-119 行（`generate_novel_background`）
- `novel_generator.py` 第 314-347 行（`generate_novel_full_background`）
- `generate_volume_background` 不构建相同的 initial_state（它直接获取 novel 数据），但第 592-637 行有独立的上下文获取逻辑（world/characters/storylines）

**重复内容：** 构建 `initial_state` dict + 从 DB 获取 world_setting/characters/storylines 注入。两处（C1 范围）代码完全相同，第三处（volume）是变体（获取相同数据但用途不同）。

**风险评估：** 低。纯内部提取，不改变函数签名和外部行为。

### C2. 章节生成逻辑重复

**已验证位置：**
- `src/core/langgraph/nodes/chapter_generation.py` 第 32-86 行：LangGraph 节点内的章节生成循环
- `novel_generator.py` 第 640-660 行（`generate_volume_background`）：prompt 构建 + LLM 调用
- `novel_generator.py` 第 783-802 行（`generate_chapters_background`）：相同逻辑

**重复内容：** 三处都执行：构建 chars_str/world_str/sl_str → 格式化 CHAPTER_GENERATION_PROMPT → 注入 style_instruction → `client.generate(prompt, max_tokens=8000)`

**差异点：**
- LangGraph 节点额外有知识图谱（KG）上下文注入和抽取
- volume/chapters 函数没有 KG 集成

**风险评估：** 中。需要保证 KG 逻辑在 LangGraph 节点中仍正常工作，同时让 volume/chapters 函数可选择性启用。

### C3. 日志混用

**已验证文件：**
| 文件 | 当前用法 |
|------|----------|
| `task_manager.py` | `import logging` / `logging.getLogger(__name__)` |
| `novel_manager.py` | `import logging` / `logging.getLogger(__name__)` |
| `outline_service.py` | `import logging` / `logging.getLogger(__name__)` |
| `storyline_service.py` | `import logging` / `logging.getLogger(__name__)` |
| `conversation_service.py` | `import logging` / `logging.getLogger(__name__)` |

`novel_generator.py` 已正确使用 `structlog`。`chapter_generation.py` 节点也用 stdlib logging。

**风险评估：** 低。structlog 兼容 stdlib logging，替换后日志输出格式统一为结构化。

### H4. generate_novel_background 和 generate_novel_full_background 共享 90% 代码

**已验证：**
- `generate_novel_background`（第 44-186 行）：构建 state → 内联 graph.astream 循环 → persist → complete
- `generate_novel_full_background`（第 292-422 行）：构建 state → 调用 `_run_langgraph_pipeline` → 额外 stages 8-13

**关键发现：** `generate_novel_full_background` 已经使用了 `_run_langgraph_pipeline` 辅助函数，但 `generate_novel_background` 没有复用它，而是内联了几乎相同的 graph streaming 逻辑。

**风险评估：** 低-中。需要确保 `generate_novel_background` 的进度百分比计算（基于 7 阶段）与 full 版本（基于 13 阶段）的差异被正确处理。

### H5. 子资源变更缺少 novel_id 归属校验

**已验证路由：**
- `PUT /{novel_id}/power-systems/{ps_id}`（第 431-443 行）：只验证 novel 存在，直接用 `ps_id` 更新
- `DELETE /{novel_id}/power-systems/{ps_id}`（第 446-455 行）：同上
- `PUT /{novel_id}/characters/{char_id}`（第 479-488 行）：同上
- `DELETE /{novel_id}/characters/{char_id}`（第 491-499 行）：同上

**安全影响：** 攻击者可通过 `/projects/novel-A/power-systems/123` 修改属于 novel-B 的力量体系。属于 IDOR（Insecure Direct Object Reference）漏洞。

**风险评估：** 低（修复本身简单），但不修复的风险高。

### H6. 3 个配置项是死代码

**已验证：**
- `TASK_STORAGE_PATH`（第 35 行）：全项目无引用
- `TASK_MAX_CONCURRENT`（第 36 行）：全项目无引用
- `TASK_CLEANUP_DAYS`（第 37 行）：全项目无引用

这些是早期 JSON 文件存储方案的遗留，现已迁移到 PostgreSQL。

**风险评估：** 极低。删除不影响任何运行时行为。

### H7. chapters 表缺少唯一约束

**已验证：** `db_models.py` Chapter 模型（第 157-178 行）无 `__table_args__`，`(novel_id, chapter_number)` 组合无唯一约束。

**风险评估：** 中。需要 alembic 迁移，且需确认现有数据无重复。

### H8. volumes 表缺少唯一约束

**已验证：** `db_models.py` Volume 模型（第 132-154 行）无 `__table_args__`，`(novel_id, volume_number)` 组合无唯一约束。

**风险评估：** 中。同 H7。

## 验收标准

| ID | 验收条件 |
|----|----------|
| C1 | `_build_initial_state` 函数存在且被 `generate_novel_background` 和 `generate_novel_full_background` 调用；原有行为不变 |

| H8 | Volume 模型有 `UniqueConstraint("novel_id", "volume_number")`；alembic 迁移文件存在 |
| 全局 | `ruff check` 通过；`mypy` 无新增错误；现有测试通过 |

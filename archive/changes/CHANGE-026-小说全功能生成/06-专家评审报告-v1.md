## 发现问题

### Blocker（必须修复）

#### B1. `generate_novel_full_background` 中任务状态管理缺陷

**文件**: `src/api/services/novel_generator.py`
**位置**: 第 357 行 `complete_task` 调用 + 第 543 行 `update_status` 调用

**问题描述**:

在 `generate_novel_full_background` 中，LagGraph 流水线（阶段 1-7）完成后立即调用了 `task_manager.complete_task(task_id, result)`（第 357 行），将任务标记为 "completed" 并持久化 LangGraph 结果。但随后阶段 8-13 通过 `_run_sub_feature` 继续执行，每个子功能结束时调用 `task_manager.update_status(task_id, "running", ...)`（第 543 行）将状态重新置为 "running"。

这导致两个问题：

1. **任务结果不完整**: `complete_task` 存储的 `task.result` 仅包含 LangGraph 产出（阶段 1-7），阶段 8-13（力量体系、大纲持久化、故事线、人物弧光、场景、对话）的结果从未被写入 `task.result`。

2. **任务终态错误**: 全部 13 阶段完成后，没有再次调用 `complete_task`。仅发布了 `EventType.COMPLETED` 事件（第 407 行），但数据库中的 `tasks.status` 停留在最后一次 `update_status(..., "running")` 调用的状态，即 "running"。这意味着任务在数据库中永远不会到达 "completed" 终态。

**建议**:
移除第 357 行的 `complete_task` 调用（或改为仅更新进度），在全部 13 阶段完成后统一调用 `complete_task`，将完整结果（包括各子功能的产出摘要）写入 `task.result`。

#### B2. `_run_sub_feature` 中导入位置导致重复 import

**文件**: `src/api/services/novel_generator.py`
**位置**: 第 452、465、490、503、516、529 行

**问题描述**:

每个子功能分支内部都有 `from src.api.services.xxx import get_xxx` 的懒加载 import。每次子功能执行时都会重新 import，虽然 Python 模块缓存会避免重复加载，但这些 import 在函数热路径中执行，违反 PEP 8 建议（import 应在文件顶部），且降低代码可读性。

**建议**:
将各子服务的 import 移到 `_run_sub_feature` 函数开头或文件顶部。当前这些 import 已存在于 `generate_novel_full_background` 和 `generate_novel_background` 之外，可以直接复用。

### Warning（建议修复）

#### W1. `_chapter_progress_callback` 硬编码阶段偏移索引

**文件**: `src/api/services/novel_generator.py`
**位置**: 第 231-232 行

```python
base = _full_generate_percentage(stage_offset + 3)  # outline_generation
span = _full_generate_percentage(stage_offset + 4) - base  # chapter_generation
```

硬编码的索引 3 和 4 对应 `STAGE_ORDER` 中的 "outline_generation" 和 "chapter_generation"。如果 `STAGE_ORDER` 列表发生变更（例如插入新阶段），这些索引将错误地指向不正确的阶段，导致进度百分比计算错误。

**建议**:
使用 `STAGE_ORDER.index("outline_generation")` 和 `STAGE_ORDER.index("chapter_generation")` 动态获取索引，或定义命名常量。

#### W2. `persist_outlines_from_result` 中 `plot_arcs` 作为总纲回退不合理

**文件**: `src/api/services/outline_service.py`
**位置**: 第 253 行

```python
master_data = result.get("master_outline") or result.get("plot_arcs")
```

在 LangGraph 的标准产出中，`plot_arcs` 通常是一个列表（情节线数组，如 `[{"name": "主线", ...}]`），而非 dict。当 `master_outline` 缺失时，`plot_arcs` 作为 truthy 值（非空列表）被选中，然后在第 255 行 `isinstance(master_data, dict)` 检查失败，进入 else 分支创建 `{"premise": str(master_data)[:500], ...}` 内容。这会导致总纲的 `premise` 字段被填入一个列表的字符串表示（如 `"[{'name': '主线'}, ...]"`），而非有意义的内容。

**建议**:
增加类型检查：`master_data = result.get("master_outline")`；如果 `master_data` 不是 dict，则将其置为 None 并跳过总纲持久化。

#### W3. `_run_sub_feature` 字符串分发难以维护

**文件**: `src/api/services/novel_generator.py`
**位置**: 第 450-540 行

使用 if/elif 字符串匹配来分发子功能，当子功能数量增长时维护成本增加，且拼写错误在运行时才能发现。

**建议**:
考虑使用 dict 映射（如 `SUB_FEATURE_HANDLERS = {"power_systems": _handle_power_systems, ...}`）或 Enum 来替代字符串分发。当前 6 个分支尚可接受，但建议在超过 10 个分支时重构。

#### W4. `generate_auto_conversation` 未校验 LLM 返回空内容

**文件**: `src/api/services/conversation_service.py`
**位置**: 第 284-303 行

`ai_content` 可能为空字符串或仅包含空白字符。当此情况发生时，仍会创建两条消息（user + assistant），其中 assistant 消息内容为空。这会导致前端展示空消息或 API 查询到无意义的对话记录。

**建议**:
添加 `if not ai_content or not ai_content.strip():` 检查，在内容为空时设置默认回退文本（如 `"当前设定较为完善，暂无补充建议。"`）。

#### W5. `generate_power_systems_ai` 缺少人物上下文

**文件**: `src/api/services/storyline_service.py`
**位置**: 第 290-340 行

生成力量体系时仅使用了 `novel_type`、`idea` 和 `world_setting`，未传入人物列表。如果人物具有特殊能力或修炼路线，力量体系应该与人物设定呼应。

**建议**:
将 `characters` 也加入 context，让 LLM 设计的力量体系与已有人物匹配。

#### W6. `persist_outlines_from_result` 中独立章节使用 volume_number=0

**文件**: `src/api/services/outline_service.py`
**位置**: 第 285 行

```python
await self.upsert_chapter_outline(novel_id, 0, ch["chapter"], ch)
```

使用 volume_number=0 作为"无归属卷"的标记。如果其他代码依赖 `volume_number > 0` 作为有效卷的判断条件，这些章节可能被遗漏或处理异常。

**建议**:
使用 `None` 替代 0 作为 volume_number，或在文档中明确约定 volume_number=0 的语义。

## 优点

1. **故障隔离设计**: `_run_sub_feature` 中每个子功能都有独立的 try/catch，单个子功能失败不会阻塞后续功能。
2. **阶段化进度推送**: 扩展的 `StageIndicator` 和新增的 `SUB_FEATURE_START`/`SUB_FEATURE_COMPLETE` 事件类型使前端可以展示更细粒度的生成进度。
3. **结果持久化分离**: `_persist_to_novel` 和 `persist_outlines_from_result` 将 LangGraph 产出与持久化逻辑解耦，职责清晰。
4. **向后兼容**: `_run_langgraph_pipeline` 提取为独立函数，原有的 `generate_novel_background` 不受影响。
5. **JSON 解析容错**: 所有 AI 生成的内容都通过 `safe_json_parse` 处理，带有 fallback 和 extract_partial 机制。
6. **小说隔离**: API 路由中所有 novel_id 都经过存在性校验，子功能操作也通过 novel_id 限定范围。

## 结论

**存在 Blocker 需修复**

B1（任务状态管理缺陷）会导致数据库中的任务永远无法到达 "completed" 终态，且任务结果不完整。这直接影响 API 消费者对任务状态的判断（如 `GET /tasks/{task_id}` 返回 status="running" 但实际已全部完成）。

B2（重复 import）虽不致命，但违反编码规范，应在修复 B1 时一并处理。

建议在修复 Blocker 后重新评审，在此之前不应合并到主分支。

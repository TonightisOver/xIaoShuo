# 需求分析 -- CHANGE-026-小说全功能生成

**日期**: 2026-05-18
**版本**: v1

## 需求概述

用户需要一键生成一部完整小说，题材为科幻/奇幻方向，建议采用 **"赛博修仙"（CyberCultivation）** 题材 -- 将传统修仙体系嵌入近未来赛博朋克世界观，科技与灵力并存，具有巨大的发挥空间。生成过程必须使用平台全部功能：LangGraph 小说生成流程、卷管理、章节管理、故事线、对话系统、大纲编辑、世界设定、角色管理、力量体系、文风系统、WebSocket 进度推送。同时需要将 DeepSeek API 配置从 `deepseek-v4-pro` + 旧 key 切换为用户提供的 `deepseek-v4-flash` + 新 key。

## 核心功能点

### F1: DeepSeek API 配置切换
- 将 `.env` 中 `DEEPSEEK_API_KEY` 改为 `sk-5ffea59097b34d12b149313dcb888b56`
- 将 `DEEPSEEK_MODEL` 从 `deepseek-v4-pro` 改为 `deepseek-v4-flash`
- 将 `DEEPSEEK_MAX_TOKENS` 从 `2000` 提升到 `16000`（`deepseek-v4-flash` 支持更长输出，章节生成需要较大 max_tokens）

### F2: 一键全功能生成流程
- 基于现有 LangGraph 流程图（idea_expansion -> world_building -> character_design -> outline_generation -> chapter_generation -> quality_check -> human_review），在其 **完成后** 自动串联执行以下子功能生成：
  - 力量体系（PowerSystem）AI 生成
  - 故事线（Storylines）AI 生成
  - 人物弧光（Character Arcs）AI 生成
  - 场景（Scenes）AI 生成
  - 三级大纲持久化（总纲 + 卷纲 + 章纲 写入 outlines 表）
  - 创作对话自动生成（可选：可生成一个"设定总结"对话记录）
  - 所有内容入库并通过 WebSocket 推送实时进度

### F3: 卷管理 & 章节管理联动
- LangGraph 的 outline_generation 节点已产出 volumes 结构（含各卷章节规划）
- `_persist_to_novel` 已实现世界设定/角色/卷/章节的持久化
- **需补齐**：将 LangGraph 产出的大纲内容同步持久化到 `outlines` 表（总纲 / 卷纲 / 章纲三级）
- **需补齐**：章节生成后自动关联到对应卷（volume_number 字段已在 Chapter 表中存在）

### F4: 故事线 & 对话系统串联
- 故事线服务已有 `generate_storylines_ai()` / `generate_arcs_ai()` / `generate_scenes_ai()` 方法，但它们目前是独立 API 端点
- 对话服务已有 `create_conversation` + LLM 问答能力
- **需补齐**：在全功能生成流程中自动调用这些生成方法，不依赖用户手动触发
- **需补齐**：生成完成后前端 NovelDetail 页面能直接展示故事线/人物弧光/场景数据（当前页面只放了跳转链接，未内联展示内容）

### F5: 世界设定 & 角色管理 & 力量体系串联
- 世界设定和角色在 LangGraph 流程中已生成并持久化，前端已有展示
- 力量体系（PowerSystem）**完全缺失**于自动化流程 -- 没有 LLM 生成方法，`_persist_to_novel` 也未处理
- **需补齐**：在 storyline_service 或 novel_manager 中增加 `generate_power_systems_ai()` 方法，并在全功能流程中调用

### F6: 文风系统
- 文风系统用于指导 LLM 生成文本的风格（句式、用词、叙事视角等）
- 当前在小说生成过程中已通过 `get_style_instruction()` 注入到各节点的 prompt 中
- 在全功能流程中应确保风格指令也注入到新增的生成步骤（力量体系描述、故事线描述等）

### F7: WebSocket 进度推送
- 当前 WebSocket 仅推送 LangGraph 7 个阶段 + 章节进度
- **需补齐**：扩展进度阶段枚举，覆盖全功能流程的所有步骤（力量体系生成、故事线生成、弧光生成、场景生成、大纲持久化、对话生成），让前端能实时看到完整进度

### F8: 前端展示完整成果
- `NovelDetail.vue` 已有 tabs 展示 overview / outlines / world / characters / chapters / storylines / conversations
- `TaskDetail.vue` 在任务完成后展示章节内容和结果
- **需补齐**：TaskDetail 完成后自动展示"查看完整项目"入口，并新增 power-systems tab 展示
- **需补齐**：NovelDetail 的 storylines tab 内联展示故事线/弧光/场景数据（当前仅放了一个跳转链接）

## 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| `deepseek-v4-flash` token 限制或格式差异 | 章节内容可能被截断或格式变更 | 增加 max_tokens 到 16000；添加响应格式兼容逻辑 |
| 全功能流程过长导致超时 | 10+ 个 LLM 调用可能超过 HTTP/WebSocket 超时 | 每个子功能独立 try-catch，失败不阻断后续；设置合理超时 |
| 多个 LLM 调用串行导致用户体验差 | 用户可能等待超过 5-10 分钟 | WebSocket 实时推送每一步进度；考虑并发调用独立子功能 |
| LLM 返回 JSON 解析失败 | 部分功能（力量体系、场景）无 fallback 值 | 为每个新增生成方法添加 fallback 逻辑 |
| 现有代码改动过多引入 bug | 影响现有功能稳定性 | 新增编排函数而非修改现有函数；全功能流程作为独立入口 |
